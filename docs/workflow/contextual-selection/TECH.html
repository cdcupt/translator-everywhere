01 Gate-1 confirmed decisions

Confirmed by the PM at Gate 1 on top of the approved Design (fixed card slot below the Recognized pane, ≤ 200 pt downward growth, 300 ms settle-then-fire, single in-flight request, skeleton shimmer, quiet inline error row, 8 s timeout, 180 ms grow / static under Reduce Motion). These three are binding for Steps 5–6:

D-1 · ⌘S swap. While a card is active, ⌘S saves the card — span as source, contextual translation as translation — and reverts to the whole-capture save the moment the card is dismissed. Implemented as a single roaming key equivalent (§02·F-7); saved through the unchanged VocabItem path (§03·4).
D-2 · Word/phrase ↔ long-span threshold. ≤ 4 whitespace tokens, or ≤ 8 graphemes for unspaced (CJK) selections — encoded as the tunable constants SelectionPolicy.maxWordTokens / maxUnspacedGraphemes (§03·3), pinned by the parameterized matrix U-01..U-11.
D-3 · Card-level read-aloud is deferred. Not in v1: the card mounts no speaker button; the pane speakers, SpeechService and PhoneticService stay untouched — part of the FR-8 frozen surface. A span-level speaker is a fast-follow.

02 Frontend — ResultPanel, selection observation & the card slot (AppKit)

The frontend adds one observed gesture (a settled selection in the Recognized pane), one new view (SelectionCardView, the only new file), and one growth behavior (the panel grows downward to make room for the card). Everything else is wiring into seams that already exist: the ResultPresenting closure-hand-off pattern, the shouldClearSelection whitelist, the phoneticGeneration stale-guard pattern, and the dropResultBody teardown sink. No new window, no popover, no event monitor.

Zero-regression argument (FR-8 / AC-8). With no selection ever made, every new code path is provably unreachable: the slot view isn't constructed until the first fire (lazy); the delegate and the .leftMouseUp hook only observe and always forward events to super; the settle task is armed only by a non-empty Recognized-pane selection; cancelOperation consumes Esc only while a card is active, otherwise super — today's Esc path bit-for-bit; the shouldClearSelection walk adds a check for a view class that isn't in the hierarchy when no card exists, so it returns identical results; ⌘S's keyEquivalent is never touched before a content state renders; and the coordinator's new members are inert until the panel invokes a hook. The only compile-visible change to existing surfaces is the added selection: parameter on ResultPresenting/showResult (defaulted to nil on the concrete method; the test spy gains the parameter) — passing nil reproduces v1.2.10 behavior exactly, which is what every pre-existing call site and test does.

03 Backend / Engine — the selection-translation service

Everything below the panel: new value types, the translateSelection orchestration inside TranslationService, the OpenAI structured-output prompt variant, the mode-threshold helper, and the notebook save path. The design rule is the same one the whole app follows: the UI never touches engines — the panel calls one hook (SelectionHooks.translate, frontend §F-3); behind it CaptureCoordinator calls one seam (Translating.translateSelection) and the panel renders whatever SelectionResult comes back. Card vs plain vs degraded is decided entirely on this side of the seam.

1 · Service architecture — types and the translateSelection flow

Fig. B1 — translateSelection is stateless and total: every route ends in a SelectionResult or a typed throw. Card vs plain vs degraded is decided here, never in the panel.

New value types live in one new file, macos/Sources/Translate/SelectionTranslation.swift. The shared contract, exact:

struct DictionaryCard: Sendable, Equatable {
    let headword: String            // ALWAYS the user's normalized span — model echo is ignored
    let translation: String         // contextual translation (required; parse fails without it)
    let partOfSpeech: String?       // lowercase English tag ("verb", "noun phrase") — nil = row omitted
    let sense: String?              // one line, target language, in-context meaning
    let example: String?            // one NEW source-language sentence, same sense
    let exampleTranslation: String? // dropped if example == nil (never orphaned)
}

enum SelectionOutput: Sendable, Equatable {
    case card(DictionaryCard)       // dictionary card variant
    case plain(String)              // long-span block AND degraded variant
}

struct SelectionResult: Sendable, Equatable {
    let output: SelectionOutput
    let servedBy: EngineKind        // .ai | .free — chip + notebook engine column
    let contextUsed: Bool           // false ⇒ "Context-free" chip (FR-5)
}

protocol Translating {              // existing seam, ONE new requirement
    func translate(text: String, pair: LanguagePair) async throws -> TranslationResult
    func translateSelection(span: String, context: String, pair: LanguagePair) async throws -> SelectionResult
}

The seam the UI consumes. Translating gains one method; the two test stubs in CaptureCoordinatorSaveTests.swift (GatedService, StubTranslating) get a mechanical one-line conformance update.

Three deliberate deviations from the whole-capture path (each keeps the selection request cheap and the approved state machine intact):
  1. No re-detect, no guard. A lone span ("scored", "攻入") cannot be reliably language-detected. The selection reuses the capture's already-resolved pair: CaptureCoordinator pins from to the capture's explicit From, else the capture's detected language, else Auto — and to = result.effectiveTo. Saves a network round-trip per selection and cannot flip the target mid-card.
  2. No runtime AI→Google fallback. The whole-capture safety net (AI failure → retry on Google) does not apply to selections: DESIGN §03's approved state machine routes network/timeout to the quiet error row with Try again. Degradation to Google is a configuration state (no key), never a runtime one — otherwise a user with AI configured would silently get context-free results exactly when they asked for context.
  3. Context is windowed, not unbounded. PRD §08 says "keep prompt small": ContextWindow.window(for:in:maxChars:) passes the full recognized text when ≤ 1500 chars, else a 1500-char window centered on the first occurrence of the span (extended outward to whitespace boundaries; prefix window if the span isn't found verbatim). Pure function, tunable constant.

Engine routing is one new method on the existing resolver — the only place that already knows preference + key + aiName:

enum SelectionRoute {
    case contextual(any SelectionSpanTranslating)   // OpenAI: span + context prompt
    case contextFree(any TranslationEngine)         // Google: span-only, degraded
}

protocol SelectionSpanTranslating: Sendable {       // adopted by OpenAIEngine ONLY
    func translateSpan(span: String, context: String,
                       pair: LanguagePair, mode: SelectionMode) async throws -> SelectionOutput
}

extension EngineResolver {
    /// Same three-branch truth table as resolve(for:): AI preferred + key + to.aiSupported
    /// → .contextual(OpenAIEngine); otherwise → .contextFree(GoogleEngine).
    func resolveSelection(for pair: LanguagePair) -> SelectionRoute
}

TranslationEngine itself is untouched — Google never needs a context parameter, so the selection capability is a separate protocol only the AI engine adopts. TranslationService keeps its existing injection pattern: the testable initializer gains an optional resolveSelection: ((LanguagePair) -> SelectionRoute)? override (mirroring detect / makeGoogleFallback) so QA can stub the route without Keychain or network.

Card mode sets response_format: {"type": "json_object"} (supported by gpt-4o-mini; the word "JSON" the mode requires is present in the system prompt). Long-span mode reuses the existing free-text transport. Both ride makeChatRequest, which gains an optional responseFormat parameter — same key, same retry/backoff, same choices[0].message.content parsing.

JSON parse + fallback strategy. static func parseCard(_ content: String, span: String) -> DictionaryCard? is a pure, unit-testable ladder:

  1. Strict: JSONDecoder straight on the content (payload struct with all-optional fields — camelCase keys match the prompt, no CodingKeys).
  2. Lenient: strip ``` fences, slice first { … last }, decode again.
  3. Hygiene (both): trim every field; empty → nil; missing/empty translation → the whole parse fails; exampleTranslation without example is dropped; headword is set from the caller's normalized span — a model-provided headword is ignored (the user's selection is the source of truth; it is also what ⌘S saves).

In translateSpan card mode: parse success → .card; parse failure with no braces in the content (model answered in prose despite JSON mode) → .plain(content) — the FR-4-shaped fallback the PRD names, no second round-trip, no double billing; parse failure with brace-like content → throw .unexpectedResponse(.ai) (never render half-parsed JSON as a translation).

3 · SelectionMode — the word/phrase ↔ long-span threshold

Fig. B3 — SelectionMode.mode(for:): a pure function of the span string alone. Grapheme counting is Swift's native String.count (extended grapheme clusters), so composed characters and emoji count as one.

enum SelectionPolicy {                                   // Gate-1 tunable constants
    static let maxWordTokens = 4                         // spaced scripts (FR-3/FR-4)
    static let maxUnspacedGraphemes = 8                  // CJK / unspaced (AC-9)
    static let requestTimeout: Duration = .seconds(8)    // DESIGN §05 "request" row
    static let maxContextChars = 1500                    // PRD §08 "keep prompt small"
    static let settleDebounce: Duration = .milliseconds(300)  // consumed by the UI layer
}

enum SelectionMode: Sendable, Equatable {
    case wordPhrase, longSpan

    static func mode(for span: String) -> SelectionMode {
        let s = SpanNormalizer.normalize(span)
        if s.unicodeScalars.contains(where: Self.isUnspacedScript) {
            let graphemes = s.filter { !$0.isWhitespace }.count      // grapheme clusters
            return graphemes <= SelectionPolicy.maxUnspacedGraphemes ? .wordPhrase : .longSpan
        }
        let tokens = s.split(whereSeparator: \.isWhitespace).count
        return tokens <= SelectionPolicy.maxWordTokens ? .wordPhrase : .longSpan
    }
}

isUnspacedScript is a named scalar-range set: Han (U+3400–4DBF, U+4E00–9FFF, U+F900–FAFF, U+20000–2EBEF), Hiragana (U+3040–309F), Katakana (U+30A0–30FF, U+31F0–31FF, U+FF66–FF9D), Thai (U+0E00–0E7F), Lao (U+0E80–0EFF), Khmer (U+1780–17FF), Myanmar (U+1000–109F). Hangul is deliberately excluded — Korean is whitespace-delimited.

4 · Notebook save path — zero schema change

Fig. B4 — Card save reuses CaptureCoordinator.save and NotebookStore.add byte-for-byte (FR-7, AC-6). Sync, export, dedupe and the FREE/AI badge all keep working because the row is indistinguishable from a whole-capture row.

Wiring: CaptureCoordinator.present(result:source:pair:) already builds the zero-argument onSave closure for the whole capture. It now additionally builds one SelectionHooks value (the frontend-declared seam type, §F-3): translate: @MainActor (String) async -> SelectionLookupOutcome — forwarding to the coordinator's translateSelectionLatest(span:context:pair:), which calls service.translateSelection(span:context:pair:) with the capture's recognized text as context and the pinned selection pair (§1, deviation 1), guards with selectionGeneration + the main generation token, and maps the return: success → .success(SelectionResult), CancellationError or a stale token → .superseded, any other throw → .failure(error) — and save: (@MainActor (_ source: String, _ translation: String, _ servedBy: EngineKind) async -> Bool)? — a thin wrapper over the same save(source:translation:from:to:kind:) with the same captured fromCode/toCode. The hooks thread through ResultPresenting.showResult as one trailing selection: SelectionHooks? parameter next to onSave/onRetranslate (signature change owned by the frontend section). ⌘S routing between header and card, and reverting to whole-capture save on dismissal, is pure UI state; the backend exposes the two closures and nothing else. The panel saves whatever variant is active: card → headword+translation, long-span/degraded → span+plain text with servedBy deciding the engine column (a degraded save records "free", so the notebook badge stays honest).

6 · What does NOT change

Frozen surface — the zero-regression guarantee (FR-8, AC-8) is enforced by construction:
  • GoogleEngine.swift — untouched. The degraded path calls the existing translate(_:) with a span-only TranslationRequest; translate_a/single has no context parameter, and we add none.
  • TranslationEngine protocol — untouched. Selection capability is the separate SelectionSpanTranslating protocol that only OpenAIEngine adopts; Google is never asked to do something it can't.
  • Whole-capture pipeline — translate(text:pair:), detect → guard → resolve → AI-fallback, the coordinator's generation token at CaptureCoordinator.swift:40, lastCapture: all byte-identical.
  • VocabItem / NotebookStore / vocab-sync server API — no new fields, no migration; a card save is an ordinary add(source:translation:from:to:engine:).
  • SettingsStore / Preferences — no new toggle (feature is passive, on by default); thresholds are compile-time constants per Gate 1.
  • Existing OpenAI translate/summarize prompts, retries, and parsing — the selection variant adds methods beside them, never edits them.

06 Risks carried forward

MED
Card latency feels slow — a second AI round-trip after selection. Held down by settle-then-fire (no mid-drag calls), single in-flight request, windowed ≤ 1500-char context, and the skeleton keeping the panel alive. Measured by M-13 against the PRD's ≤ ~3 s typical target; if it misses, first lever is prompt size, not architecture.
MED
Structured-output flakiness — the model returns malformed card JSON. Contained by the parseCard ladder: strict → lenient → prose-as-.plain fallback (FR-4 shape, no second billing); brace-like garbage throws rather than rendering half-parsed JSON. Pinned by U-22..U-27 and I-01.
LOW
Selection UX conflicts — the card must not fight click-outside collapse, read-aloud, or phonetics. One dismissal sink, the shouldClearSelection whitelist entry, and Esc consumed only while a card is active keep the affordances separated. Pinned by P-01, I-13, M-05.
LOW
Cost creep — every selection is a paid call on the user's key. Fire only after a settled mouse-up/keyboard selection; identical-span dedupe; never mid-drag. Pinned by P-02 and the Fig. F2 guards.
LOW
Protocol-change ripple — Translating and showResult(selection:) touch existing conformers. Bounded by Table Q6's allowed-changes list; any compile error outside that table is a design-drift signal to stop and raise, not patch around.
LOW
Serialized-bundle flakiness — new async tests join a non-parallel bundle. The gate runs the suite twice (Fig. Q4); races are built on the deterministic GatedService continuation pattern, never sleeps.
