01 Problem

Zone capture today is all-or-nothing: whatever text Vision recognizes in the dragged region is translated as one block. But the reason a user captures a zone is often a single word or phrase inside it they don't understand. Their two current options are both bad:

Re-capture a tighter zone around just that word — fiddly, and it throws away the sentence, so the translation loses the context that determines the word's meaning. Or read the whole-block translation and try to reverse-engineer which output words correspond to the word they cared about — unreliable across EN⇄zh word-order differences.

The insight: the app already has the full context (the recognized text) and already renders it in a selectable pane. What's missing is the bridge — act on a selection, and let the surrounding text disambiguate it.

02 Users & primary scenario

Language learners and bilingual readers (EN⇄zh first) reading content they can't select natively — video subtitles, images, PDFs, apps with non-selectable text. The primary scenario:

04 Scope

In scope

  • Text selection in the Recognized pane triggers translation of just that span
  • Full recognized text passed as disambiguating context (AI engine)
  • Mini dictionary card for word / short-phrase selections
  • Contextual translation block for longer selections
  • Graceful degradation when only Google engine is configured
  • ⌘S saves an active card to the Notebook (existing fields)
  • Stale-result safety: selection requests respect the existing generation token

Out of scope

  • On-screen Live-Text-style overlay over the captured region (possible later phase)
  • Re-running OCR on a sub-region of the zone
  • Selection in the Translation pane (reverse lookup)
  • VocabItem / server vocab-sync schema changes (no new context field)
  • New Preferences toggle — the feature is passive and selection-driven, on by default
  • iOS / other platforms

05 Functional requirements

FR-1 · Select-to-translate

Selecting text in the Recognized pane and releasing the mouse triggers a translation of exactly the selected span, debounced, with at most one in-flight selection request (a newer selection cancels/supersedes an older one).

FR-2 · Context-aware prompt

When the AI engine is available, the request carries the full recognized text as context plus the selected span, and instructs the model to translate the span as used in that context. The whole-text translation already on screen is not replaced.

FR-3 · Mini dictionary card

For a selection of ≤ N words (default 4, tunable constant), the result renders as a card with: contextual translation (prominent), part of speech, one-line in-context sense, and one usage example pair. Fields the model can't fill are omitted, never faked.

FR-4 · Long-span mode

For selections longer than N words, the result is a contextual translation of the span only (no POS/example fields).

FR-5 · Degraded mode

With no AI engine configured, the selection is translated by Google without context and the result is visibly labeled as context-free (subtle, not alarming). The feature never errors out solely because context is unavailable.

FR-6 · Dismissal & lifecycle

The card dismisses on click-outside, Esc, selection collapse, retranslate (language change), and new capture. A stale card never survives a content change (reuse the generation-token guard).

FR-7 · Notebook save

With a card active, ⌘S saves the selected span as source and the contextual translation as translation, using the existing VocabItem fields and sync path unchanged.

FR-8 · Zero regression

With no selection made, every existing flow (capture, whole-text translate, retranslate, read-aloud, phonetics, save, history) behaves identically to v1.2.10.

07 Constraints & key decisions

Context requires the AI engine. Translation is fully client-side: Google's keyless endpoint (translate_a/single) accepts no context, so contextual results ride the OpenAI engine (gpt-4o-mini, BYOK). Degraded mode (FR-5) covers Google-only users.
OCR output is a flat string. OCRService discards Vision bounding boxes; per-word geometry doesn't exist downstream. Selection therefore comes from the NSTextView in the panel — which also makes zh selections work without a tokenizer. This is what makes the result-window approach cheap and the overlay approach expensive (hence deferred).
Structured card output. The card needs machine-readable fields (translation / POS / sense / example) from the model — a structured-output prompt variant in OpenAIEngine, with a defined fallback if parsing fails (degrade to plain contextual translation, FR-4 shape).
No new persistence. Notebook save reuses VocabItem as-is; no SwiftData migration, no server API change. If context-in-notebook proves valuable later, it's its own feature.

Platform: macOS 14+, AppKit result panel, xcodegen project, XCTest (non-parallel). New logic goes behind the existing protocol seams (Translating etc.) for testability.

08 Risks

MED
Card latency feels slow — a second round-trip to OpenAI after selection. Mitigation: debounce + spinner-in-card, keep prompt small, single in-flight request; acceptance target ≤ ~3 s typical.
MED
Structured output parsing flakiness — model returns malformed fields. Mitigation: JSON-mode / strict prompt + parse fallback to plain contextual translation (never an error dialog).
LOW
Selection UX conflicts — the panel already collapses selections on outside-click; read-aloud/phonetics also act on pane text. Design must reconcile the affordances so they don't fight.
LOW
Cost creep — every selection is a paid AI call (user's own key). Mitigation: explicit mouse-up trigger + debounce; no auto-fire while dragging.

09 Success criteria

Ship when: all acceptance criteria pass; the full existing test suite stays green (zero regression); QA test plan green; the beta panel signs off (two lens testers ship + all-PASS feature coverage matrix). Qualitative bar: selecting a word feels like a native feature of the panel — one gesture, one card, no mode to enter.
