Translator Everywhere — app + vocabulary + sync

A native menu-bar macOS app: press a global hotkey, drag a box over any text on screen, and read the translation instantly. Every capture is saved to a vocabulary notebook the user can review, search, export, and summarize. The notebook lives on-device by default; signing in with Google or Apple syncs it across Macs via our own VPS. Downloaded from GitHub, installed by drag-to-Applications, hotkey configured inside the app.

Fig 1 — The whole product on one screen: the capture loop saves every word to a notebook that is local-first by default and cloud-synced when you sign in. Three pillars: easy install · easy hotkey · free to start.

1Problem & why now

The te CLI already does screen-capture translation, but it serves only a technical user on one Mac, throws every translation away, and needs the author's paid key. We want a product anyone can download and use — and one that turns each "what does this mean?" moment into durable vocabulary the user can come back to. Two jobs:

  • Translate anything on screen, instantly, with one gesture — videos, web pages, PDFs, non-selectable UI.
  • Remember it — collect captured words/phrases into a notebook to review and summarize later, on one Mac or across several.

2Users & the core jobs

Primary — the language learner

Reads/watches in a second language. Wants one-gesture translation and a growing vocab list to study. May use one Mac (local) or several (sync).

Secondary — the privacy-minded user

Wants the tool but no account and nothing on anyone's server. Local-only must be a first-class, complete experience.

Power user (Erik)

Wants OpenAI-quality translation, configurable hotkey, sync across machines, and export.

4System architecture

Fig 2 — Three-tier but privacy-scoped: OCR is on-device, translation goes direct to the engine, and our VPS only ever sees text vocabulary rows for signed-in users. Local-only users never touch the server.

5Data, notebook & sync

  • The notebook row: source text, translation, source/target language, timestamp, engine used, optional tag/note, a sync id + updated-at. Images are not stored.
  • Local-first: the app is fully functional with the local store alone. No account, no network for the core loop + notebook.
  • Sync model: on sign-in, the local notebook is pushed up and the cloud copy pulled down; ongoing changes sync both ways with last-write-wins per row (updated-at). Sign-out keeps the local copy.
  • Summarize: a "make a study list / summary" action runs over the notebook through the active translation engine (on the client) — e.g. group by theme, add example sentences. No server AI.
  • Export: CSV + Markdown of the notebook, any time, local or signed-in.
Privacy boundary (non-negotiable): the captured image is OCR'd on-device and discarded. Only recognized text is translated (direct to the engine) and, for signed-in users only, stored as vocab rows on our VPS. We publish a short privacy policy and support account/data deletion.

6Accounts & the signing consequence Gate-1 sign-off

⚑ Coordinator note — adding accounts forces real code-signing

Sign in with Apple requires a code-signed app with the Apple Sign-In entitlement (an unsigned/ad-hoc app can't use it). So merging cloud sync into v1 means v1 must ship Developer-ID signed + notarized (using the existing Apple Developer account). Upside: zero Gatekeeper friction. This replaces the earlier "unsigned v1" plan.

Providers: recommend Google + Apple (Apple is expected on Apple platforms; Google is broad). Alternative: Google-only avoids the Apple entitlement but is a weaker fit on macOS. Needs your nod at Gate 1.

7Engine decision (unchanged) Gate-1 sign-off

Translation engine stays "Both, switchable" with a FREE default: Apple Vision OCR + Google Translate (free, no key) by default; an optional OpenAI key (stored in Keychain) for higher-quality AI translation. The backend is not involved in translation either way.

9Scope (v1 = everything)

In — v1 must

  • Menu-bar app, app-registered global hotkey.
  • Region capture → Vision OCR → translate → result popup + clipboard.
  • Vocabulary notebook: auto-save, list, search, delete, export (CSV/MD), summarize.
  • Preferences: hotkey, engine (free/OpenAI key), default direction, account.
  • Accounts: Sign in with Google + Apple; cloud sync of the notebook.
  • Backend service on BWH: OAuth, sync API, Postgres; subdomain + TLS.
  • First-run onboarding incl. Screen Recording permission + relaunch.
  • Secrets in Keychain; signed + notarized .dmg on GitHub Releases.
  • Short privacy policy + account/data deletion.

Later v2+

  • Launch-at-login; translate-clipboard / translate-selection items.
  • In-place overlay instead of a dialog; spaced-repetition study mode.
  • Auto-update; iOS companion.

Out not now

  • App Store / sandboxed distribution.
  • Any AI or translation on our server — stays client-side.
  • 9relay as the backend — this gets its own service.
  • Storing captured images anywhere.
  • Windows / Linux.

10Constraints & non-negotiables

  • Privacy: OCR on-device; images discarded; only text translated (direct to engine) and, for signed-in users, synced. No telemetry. Privacy policy + deletion.
  • Secrets: OpenAI key + auth tokens only in Keychain; never committed/logged. No bundled author key. Backend secrets in an env file on the VPS, never in the repo.
  • Backend: its own service (not 9relay), deployed on BWH per the multi-tenant rules — own DB+role, own port, own Caddy snippet, never disrupting 9relay/BillMind/etc. Proposed subdomain api.translator.daichenlab.com.
  • Signing: Developer-ID signed + notarized (required by Sign in with Apple).
  • Repo: cdcupt/translator-everywhere; app under /macos, backend under /server; the te CLI untouched.
  • Build: macOS app via DEVELOPER_DIR + xcodegen (Xcode 26.5). Swift + SwiftUI/AppKit. Backend: a small service (language TBD in Tech Doc) on the shared Postgres pattern.
