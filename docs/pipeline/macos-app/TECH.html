1Backend stack choice

This is a small CRUD + auth sync service, not a heavy application: a handful of JSON endpoints, one Postgres database, JWT verification, and OAuth/OIDC token validation. The right stack is boring, fast to deploy, light on RAM, and a clean fit with the existing BWH apps (several of which ship as containers built via GitHub Actions → GHCR). It must coexist politely on a shared box — small memory footprint, single static binary preferred.

Recommended — Go + chi + sqlc + goose, shipped as a single container

  • Language: Go 1.22+. Compiles to one static binary; tiny container (~15–25 MB), low idle RAM (tens of MB) — ideal for a shared VPS that already runs 9relay, BillMind, scenario-english.
  • Web framework: chi — a stdlib-compatible router with middleware (logging, recover, real-IP, rate-limit). Minimal surface, no ORM magic. Handlers are plain net/http.
  • DB access: pgx driver + sqlc (compile SQL → type-safe Go). No runtime ORM; queries are reviewable SQL.
  • Migrations: goose — plain, ordered .sql files run on deploy. The shared-Postgres pattern in BillMind/scenario-english already uses ordered SQL migrations.
  • Auth libs: golang-jwt/jwt for our session tokens; coreos/go-oidc + jwks caching to verify Apple & Google identity tokens (handles key rotation).
  • Container: multi-stage Dockerfile, built in CI (GHA) and pushed to GHCR, exactly like the other BWH apps; pulled + run on the VPS via compose.

One-line rationale: a single tiny static Go binary is the cheapest, most operationally consistent way to add an auth+sync JSON API to an already-crowded shared VPS.

Top alternative — and why not (for v1)

Go is the recommendation; Node + Fastify + Drizzle is the named fallback if the team prefers a JS toolchain. Either way the architecture below is identical — only the handler language changes.

2Data model

Two tables. users is identity only (which provider, which subject, an email for display). vocab_items is the synced notebook row — a near-mirror of the on-device row, plus the fields that make sync correct: a client-generated UUID for idempotent upsert, an updated_at for last-write-wins, and a deleted tombstone so deletes propagate instead of resurrecting.

Fig 2 — ER diagram. users is keyed by the provider identity (Apple/Google subject); vocab_items belongs to one user and is keyed for sync by (user_id, client_uuid). The deleted tombstone + updated_at drive last-write-wins merge and idempotent upsert.

Constraints & indexes

  • UNIQUE (provider, provider_subject) on users — one account per provider identity; the upsert key at sign-in.
  • UNIQUE (user_id, client_uuid) on vocab_items — idempotent upsert target; a re-pushed row updates rather than duplicates.
  • INDEX (user_id, updated_at) — powers GET /vocab?since= (the only hot query) efficiently.
  • FK vocab_items.user_id → users.id with ON DELETE CASCADE — supports one-statement account deletion.

3Auth flows

Both providers follow the same shape: the app drives the native sign-in, gets a provider identity token, and POSTs it to our /auth/<provider> endpoint. The server verifies that token against the provider's public keys (handling key rotation via a cached JWKS), upserts the user, and returns our own session JWT. From then on the app authenticates to the sync API with our token — the provider is out of the loop. Sign-out is purely client-side token disposal plus an optional server revocation note.

Sign in with Apple — native

Fig 3 — Sign in with Apple. The app uses the native ASAuthorizationController; the server verifies the Apple identity JWT against Apple's JWKS (cached, auto-refreshed for key rotation), checks iss/aud(our bundle id)/exp, upserts the user by (apple, sub), and returns our session JWT. Apple may relay/withhold email — we store whatever is provided, for display only.

Sign in with Google — OAuth 2.0 + PKCE

Fig 4 — Sign in with Google (Authorization Code + PKCE, the correct flow for a native app with no safe client secret). The app gets the id_token and hands it to us; we verify it against Google's JWKS (iss/aud/exp), upsert by (google, sub), and issue our session JWT. The Google token-exchange can also run server-side if we prefer to keep the exchange off-device; both are noted.
Session token lifecycle. Our session JWT is short-ish lived (e.g. access ~1h) and paired with a longer-lived refresh token (e.g. 30–90 days) so the app rarely re-runs the provider dance. POST /auth/refresh swaps a valid refresh token for a fresh access JWT. Sign-out deletes both tokens from the Keychain and calls POST /auth/signout (best-effort) to drop the server-side refresh record; the local notebook is untouched (per DESIGN §2e). Access JWTs are signed with a server secret (HS256) or a server keypair (RS256) — secret/keys live in the VPS env file.

4Sync API

The sync API is deliberately tiny: exchange an identity token for a session, pull rows changed since a cursor, push a batch of local rows, delete the account. Everything is authenticated with the session JWT in an Authorization: Bearer header. The merge rule is last-write-wins per row keyed on updated_at, with tombstones carrying deletes.

Method · pathAuthPurpose
POST /auth/applepublicVerify Apple identity token → upsert user → return session JWT + refresh token.
POST /auth/googlepublicVerify Google id_token → upsert user → return session JWT + refresh token.
POST /auth/refreshrefresh tokenExchange a valid refresh token for a new access JWT.
POST /auth/signoutBearer JWTBest-effort revoke the refresh token server-side. (Local copy kept.)
GET /vocab?since=<ts>Bearer JWTReturn all this user's rows (incl. tombstones) with updated_at > since. Pull half of sync.
POST /vocabBearer JWTBatch upsert of local rows by (user_id, client_uuid); per-row last-write-wins. Push half of sync.
DELETE /accountBearer JWTDelete the user + all rows (CASCADE). Backs "Delete account & cloud data".
GET /healthzpublicLiveness/readiness (DB ping) for the status page & deploy smoke-test.

The merge — last-write-wins per row

Fig 5 — A sync cycle. ① Push dirty local rows (batch upsert, idempotent on client_uuid); ② Pull everything changed since the last cursor (including tombstones); ③ Merge locally, keeping the newer updated_at per row and applying deletes. The cursor advances to the server's clock so the next cycle is incremental. Sync is best-effort & background — it never blocks the capture popup.
  • Last-write-wins per row: on both sides, the copy with the greater updated_at wins. Simple, deterministic, good enough for a personal notebook (no field-level merge needed).
  • Tombstones: a delete is a row with deleted=true and a bumped updated_at — it travels through sync so other devices remove it too. A periodic job purges tombstones older than e.g. 90 days.
  • Idempotency: (user_id, client_uuid) means a retried push never duplicates; the same row just re-upserts.
  • Cursor: the app stores last_sync = server_time from the pull and sends it as ?since= next time — incremental, cheap, single-index scan.

5Deployment on BWH

The service is a good multi-tenant citizen on the shared BWH box. It adds itself through its own Caddy snippet (one import line into the shared 9relay-caddy config), binds a free localhost port, and uses its own database + role on the shared Postgres. It never edits another app's config and never touches 9relay / BillMind / scenario-english.

Fig 6 — Infra. Caddy on :443 is the single shared edge; it imports our own translator.caddy snippet, which reverse-proxies api.translator.daichenlab.com to our container on a free localhost port. The container talks to its own DB + role on the shared Postgres. Secrets live only in an env file on the VPS.

Own DB+role vs own Postgres — recommendation

Recommend: own DATABASE + ROLE on the shared Postgres

Create translator_everywhere as a dedicated database with a dedicated role (te_app) on the existing shared Postgres — the same pattern BillMind and scenario-english already use successfully. Two tiny tables and a personal-scale workload don't justify a second Postgres process competing for RAM on a crowded box. A separate role gives clean credential isolation and per-DB backups; logical separation is sufficient here. Only stand up a dedicated Postgres if a future requirement (heavy load, separate backup/retention policy, or stricter isolation) demands it — Diecast Vault does this, but this service does not need it.

6Security & privacy

Token storage

Session JWT + refresh token live only in the macOS Keychain (alongside the OpenAI key) — never on disk in cleartext, never logged. Server holds only a hashed/opaque refresh-token record.

Transport

TLS everywhere (Caddy auto-cert). HSTS on the subdomain. Authorization: Bearer only over HTTPS. No tokens in URLs or query strings.

Token verification

Apple/Google identity tokens verified against the provider JWKS (cached, auto-rotated): signature, iss, aud (our client/bundle id), exp. Our own JWT verified on every sync request.

Rate limiting

Per-IP and per-user limits (chi middleware) on /auth/* and /vocab — blunts credential stuffing and abusive batch pushes. Sensible batch-size cap on POST /vocab.

Account & data deletion

DELETE /account cascades to wipe all rows + the user; backs the DESIGN "Delete account & cloud data" control. Tombstone GC keeps the table from growing unbounded.

Text-only guarantee

The schema has no image column and no binary blob — it is structurally impossible to store a capture image. The server has no OCR, no translation, no AI. Enforced by design, not policy.

Input validation. Every request body is validated at the boundary: required fields, length caps on source_text/translation, allowed engine/lang values, batch-size limit. Reject malformed payloads with a clear 4xx; never trust client-supplied user_id (always derive it from the verified JWT sub).

The Google free-translate endpoint is NOT a server concern

Translation runs client-side, direct to Google's free endpoint — our backend never proxies or sees it. Any abuse/rate-limit risk of that free endpoint is a client matter (the app should back off / surface the "engine-down" error from DESIGN §4), not a server-side liability. Our server can't be blamed for it because our server isn't in that path. This keeps the backend's attack surface to just auth + a two-table sync store.

8.3 · Local persistence — SwiftData schema & offline-first

The local store is SwiftData (Core Data is the fallback; the schema is identical either way). It mirrors the server vocab_items row so a synced row maps 1:1, and adds the small bookkeeping a client needs: a stable clientUUID generated on-device at row creation (the join key for sync — it must survive restarts), a local dirty flag for the offline change queue, and a deleted tombstone so deletes propagate instead of resurrecting. The notebook is fully functional with this store alone — no account, no network.

Fig 9 — Local VocabItem ↔ server vocab_items. Every field maps 1:1 except the client-only dirty flag (the push queue) and the server-only id/user_id (the server derives user_id from the JWT — the client never sends it). clientUUID is @Attribute(.unique), generated once at creation and persisted, so it is the stable join key across restarts and Macs.
  • Offline-first: create/edit/delete write to SwiftData immediately and set dirty = true; the UI updates from the store at once. A delete sets deleted = true + bumps updatedAt (tombstone) rather than hard-deleting, so the delete can sync.
  • The change queue is the store: there is no separate queue table — "the queue" is simply predicate: dirty == true. On the next sync the SyncClient pushes those rows and clears dirty on success.
  • Reconcile = last-write-wins per row (matches §2/§4): on pull, for each incoming row, if its updatedAt is newer than the local copy, overwrite (and apply tombstones); otherwise keep local. Deterministic, no field-level merge — correct for a personal notebook.
  • Local-only users never set up sync: dirty just stays set harmlessly and is never read. Signing in later pushes the whole backlog in one batch.

8.4 · Translation engine protocol

Translation never goes through our server — the engine talks directly to Google or OpenAI (§6 / PRD §5). A single protocol makes the two interchangeable; the active engine is chosen in Preferences ▸ Engine and resolved at call time, so switching is a one-line swap and Summarize reuses the same path.

protocol TranslationEngine {
  var badge: EngineBadge { get }            // .free | .ai → drives the popup/notebook badge
  func translate(_ text: String, to: Lang, from: Lang?) async throws -> Translation
  func summarize(_ rows: [VocabItem]) async throws -> StudyList
}
struct GoogleEngine: TranslationEngine   // free public endpoint · no key · default · badge=.free
struct OpenAIEngine: TranslationEngine   // user key from Keychain · reuses the `te` script prompts · badge=.ai

// resolved from SettingsStore at call time — caller never branches on engine type
func activeEngine() -> TranslationEngine {
  settings.engine == .openai && keychain.openAIKey != nil ? OpenAIEngine(...) : GoogleEngine()
}
  • Default is free & keyless — GoogleEngine. OpenAIEngine is selected only when the user both chooses OpenAI and has a valid Keychain key; otherwise we fall back to free (DESIGN §2d "always free" override).
  • Prompt reuse: OpenAIEngine ports the existing te CLI prompts verbatim, so AI translation quality matches the tool the PRD is built on.
  • Error mapping: each engine throws a typed TranslateError (offline, engineDown, badKey, quota, unknown) that maps directly onto the DESIGN §4 error copy.
  • Summarize is just summarize(rows:) on the same active engine — client-side, no server AI (PRD §5); free yields a grouped list, OpenAI yields richer example sentences (DESIGN §2c).

8.5 · Sync client — speaking the §7 contract

The SyncClient is an actor wrapping URLSession. It attaches Authorization: Bearer <session_jwt> to every /vocab call, runs a push-then-pull cycle, and persists a last_sync cursor (server clock) in SettingsStore. It is background and best-effort — failures are retried later and never surface as blocking errors; the notebook keeps working offline.

TriggerWhat runs
On sign-inFull push of the entire local notebook (all rows as a batch POST /vocab), then GET /vocab?since=0 to merge anything already in the cloud.
On change (debounced)After a row is saved/edited/deleted and the store goes dirty, a ~2 s debounce coalesces edits, then pushes the dirty batch + pulls the delta.
On app foregroundA pull (GET /vocab?since=last_sync) to pick up changes made on another Mac, followed by a push if anything is dirty.
On token expiry (401)POST /auth/refresh for a new access JWT, then retry once. If refresh fails → drop to signed-out, chip shows the local-only state; nothing is lost.
  • Push: POST /vocab { items: [rows where dirty] } — idempotent on (user, client_uuid); on 2xx, clear dirty for the applied rows. Honor any conflicts[] the server returns (server row was newer → adopt it locally).
  • Pull: GET /vocab?since=last_sync → apply each row by last-write-wins (incl. tombstones), then set last_sync = server_time so the next cycle is incremental.
  • Conflict handling = LWW per row, identical on both sides (§4): the greater updatedAt wins; no merge UI ever shown. Tombstones win the same way.
  • Chip state is derived from SyncClient: Local only (signed out) · Syncing… (in-flight) · Synced ✓ · email (idle & clean) · Offline (signed in, last attempt failed) — exactly DESIGN §2c.
  • Batch cap: respects the server's batch-size limit (§6) by chunking large initial pushes.

8.6 · Auth UI flow

Both providers end the same way: get a provider identity token on-device, POST it to the §7 auth endpoint, receive our session_jwt + refresh_token, store both in the Keychain, and flip the Account tab to signed-in.

8.7 · Concurrency model

Off the main actor

CaptureCoordinator, SyncClient, and engine calls are actors / async; the screencapture Process, Vision request, and all URLSession work run off main. Capturing/OCR/network never block the UI run-loop.

On the main actor

Panel presentation (NSApp.activate, become-key), clipboard write, and any view that observes NotebookStore are @MainActor. SwiftData writes happen on the store's context, then UI updates land on main via the model container.

Hand-offs

The actor produces a value-type Translation; the UI hop is a single await MainActor.run { present(...) }. No shared mutable state crosses actors — only Sendable value types.

Best-effort sync

Sync is detached from the capture task: a save returns immediately and a separate Task on the SyncClient does the network. A sync failure can never fail a capture or a save.

9QA test plan

How we prove the build matches the PRD before Gate-3 and the two beta-testers. The plan splits cleanly along the privacy boundary: the macOS app (Swift — capture, OCR, translate, notebook, sync client, auth, permissions) and the backend (Go — auth verify, the two-table sync store). Most logic is covered by fast unit/integration tests that run in CI; a small ring of E2E/manual scenarios covers what only a real Mac, a real provider account, and a second machine can verify. Every test traces back to a PRD goal — the G1–G8 acceptance matrix is the contract.

9.1 · Test strategy — the pyramid, both halves

A wide base of cheap unit tests, a thinner band of integration tests (backend against a real ephemeral Postgres; app stores against an in-memory model container), and a deliberately small cap of E2E/manual scenarios. The two columns are tested independently and meet only at the §7 contract, which is pinned by a shared fixture so app and server can't drift.

Tooling. App: XCTest via swift test / xcodebuild test on a macOS destination + an in-memory SwiftData ModelContainer; network mocked with a custom URLProtocol; the Keychain exercised in a test keychain so CI never touches the login keychain. Backend: go test ./... with -race and -cover; integration tests spin an ephemeral Postgres (dockertest/testcontainers) and run goose up first, so migrations are tested on every run.

9.3 · Unit & integration — the backend

Unit tests cover the pure resolvers and token logic with mocked providers; integration tests run handlers against an ephemeral Postgres with migrations applied, so the SQL and constraints are exercised for real.

TestLayerWhat it proves
Auth verify — Apple/Googleunit (mocked JWKS)A valid provider identity token verifies (sig · iss · aud=our bundle/client id · exp) and upserts the user by (provider, subject); a tampered/expired/wrong-aud token is rejected with 401.
JWKS key rotationunitA token signed with a freshly-rotated provider key still verifies after the cached JWKS refreshes; a token with an unknown kid triggers a re-fetch then accept/reject correctly.

9.5 · Edge cases

Capture region

Empty (zero-area drag) → treated like cancel, no save. Huge region (full screen, dense text) → OCR still completes, panel readable, no truncation crash. Non-text image (photo/logo, no glyphs) → clean no-text-found, nothing saved.

Network

Drop mid-translate → error panel + clipboard fallback, Retry works. Google rate-limit / back-off → "engine-down" copy, suggest OpenAI; client backs off (server is never in this path — §6).

Auth / token

Expired access token → transparent refresh, sync resumes. Expired/revoked refresh → quietly drop to local-only, surface a gentle "sign in again". Apple email-withheld → account still works.

Sync conflicts

Duplicate client_uuid (retry/two devices) → one row, idempotent. Conflicting edits to the same row on A and B → newer updated_at wins deterministically; the loser is reported in conflicts[] and reconciled, never lost silently into a dupe.

9.8 · Performance

Free-path latency budget (G4)

capture → result < ~3 s on the free path (Vision OCR on-device + Google translate), measured on a representative paragraph. Instrumented by timestamping coordinator stages (capture done · OCR done · translate done · panel up) and asserting the wall-clock budget in a manual perf pass on real hardware. The panel must appear the instant a translation exists; auto-save and background sync run after the panel is up and never count against this budget (§8.2). Backend: /vocab push/pull stays comfortably sub-second for a personal-scale notebook (single-index since scan).

9.9 · CI / automation & the green-before-merge rule

Fig 15 — CI vs manual. Automated: swift test, go test -race -cover, the migration check, secret-scan, and signing verification — all on every push, with an 80% coverage floor. Manual (real Mac / real accounts / second machine): capture, permission+relaunch, sign-in, two-Mac sync, Gatekeeper, and the perf pass. Green-before-merge: no PR merges while any automated job is red; the manual pass + the two beta-testers gate the release itself.

Open items needing a decision (QA flags for the Coordinator)

  • Test provider accounts. The Apple + Google sign-in E2E and the two-Mac sync scenarios need a real test Apple ID and test Google account (and the app registered in both consoles with the right bundle/client id + redirect). Until those exist, G6 and the auth happy-path are unit/integration-only (mocked JWKS) — fully covered logically but not end-to-end.
  • Second Mac for sync. G6's round-trip needs two machines (or a second user session). A single CI box can't prove cross-device sync — this stays a beta-tester task.
  • Google free-endpoint variability. The free Google endpoint can rate-limit unpredictably; its E2E is best-effort and the back-off behaviour is what we actually assert, not a fixed latency.
