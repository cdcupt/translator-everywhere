★At a glance — screen map

Fig 1 — One menu-bar item is the whole app. Its menu now opens six surfaces. Two are new for rev 2: the Vocabulary Notebook (its own resizable window, reached via "Open Notebook" ⌃⌥N) and the Account preferences tab. The Result popup is what users see most — it now auto-saves every capture to the notebook; Onboarding still runs once.

1Interaction model — the capture flow as states

One hotkey press drives a short-lived state machine. The user only ever sees three things: the native crosshair, then (briefly) nothing while we work, then the result panel. Every non-happy path resolves into a clear, dismissible panel — never a silent dead end, except a deliberate Esc-cancel, which returns to idle with no UI at all.

★ Auto-save & background sync (rev 2)

Every successful translation is auto-saved to the Vocabulary Notebook — the user never has to press "save." On the result state we surface this with a quiet "★ Saved to Notebook" affordance sitting beside "✓ Copied", so it reassures without competing with the translation. No-text and error states save nothing. When the user is signed in, the new row also syncs in the background (the sync client pushes it up); when local-only, it just lands in the on-device store. Sync never blocks the popup — the result is shown instantly and syncing happens after dismissal. The notebook window's toolbar reflects the live sync state (Local only / Syncing… / Synced ✓ / Offline).

Fig 2 — The capture state machine. Happy path runs left→right then drops to result. Esc during selection returns to idle silently (dashed). No-text and error are recoverable panels. The permission gate diverts to onboarding step 2 before any capture starts.

⚑ The relaunch problem — handled in onboarding

Screen Recording permission only takes effect after the app restarts. So onboarding never tries to capture while ungranted. Step 2 detects the permission, opens System Settings to the right pane, and offers a "Quit & Reopen" button that relaunches the app for the user. On the fresh launch we re-check: granted → jump straight to step 3 ("Done — try it"); still not granted → stay on step 2 with a gentle "Still not enabled" hint. The hotkey is also live before permission is granted: pressing it while ungranted reopens onboarding step 2 rather than failing silently.

2aMenu-bar menu

The status item is a monochrome template glyph that adapts to light/dark menu bars automatically. Click it → this menu. It is the only always-available entry point.

Fig 3 — Menu-bar menu (rev 2). The "Capture & Translate" row echoes the user's current hotkey so the binding is always discoverable. "Translate Clipboard" is the cheap should from PRD §7. New "Open Notebook…" (⌃⌥N) opens the Vocabulary Notebook window, set off in its own group between the capture actions and Preferences.

Behaviour notes. The hotkey label is data-bound to the recorded shortcut — change it in Preferences and the menu updates. If Screen Recording isn't granted yet, "Capture & Translate" still works but routes to onboarding step 2. "Open Notebook…" brings the notebook window forward (creating it on first use) and always works — it needs no permission and no account. "About" opens the same content as the Preferences About tab in a compact standalone panel.

2bResult popup

A small, focused NSPanel that appears near the captured region (or screen-centred as a fallback). Because agent apps launch unfocused, the panel explicitly becomes key and orders front so Esc and ⌘C work immediately. The translation is the hero; the source text is present but quiet.

Fig 4 — Result popup, success. Translation in the serif display face; recognized source dim below; "Copied ✓" confirms the silent clipboard write and a quiet "★ Saved to Notebook" (rev 2) confirms the auto-save; the engine badge reads FREE or AI.

Variants — no-text and error

Fig 5 — Left: no-text-found (recoverable, Try Again re-arms the crosshair). Right: error — message names the real cause (offline / engine down / bad key) and offers Retry plus, for key/engine problems, Preferences. The recognized text is still copied where possible.

2cVocabulary Notebook window new · rev 2

The notebook is the app's second real window — a standalone, resizable document-style window (not a popover), opened from "Open Notebook…" (⌃⌥N). It is where every capture lives: a reverse-chronological table the user can search, multi-select, delete, export, and summarize. It works fully local-only; a toolbar sync chip reflects cloud state when the user is signed in. The window is deliberately quiet and information-dense — Fraunces only for the empty-state hero, otherwise the system face so it reads like a real macOS list view.

Fig 5 — Vocabulary Notebook window. A reverse-chronological table: source, translation, language pair, FREE/AI engine badge, optional tag, date. Toolbar holds a search field, a sync-status chip, Export ▾ (CSV / Markdown), and Summarize ★. Rows are multi-selectable (checkboxes); the footer shows the count and a destructive Delete selected.

Sync-status chip — the four states

The chip in the notebook toolbar is the single place the user reads "where does my data live?" It is calm by default and only colours up when it has something to say.

Fig 6 — Sync-status chip states. Local only (grey, the default), Syncing… (amber, transient), Synced ✓ (email) (blue, signed-in & current), Offline (red, signed-in but unreachable — local store still works, will sync when back). Clicking the chip jumps to Preferences ▸ Account.

Summarize → study list

The Summarize ★ action runs client-side over the selected rows (or all rows if none selected), sending them through the active translation engine — exactly like a capture, no server AI. The result is a generated study list: items grouped by theme with a short example sentence each. It opens in a sheet over the notebook with Copy and Export. With the Free engine it produces a simple grouped list; with the OpenAI engine it produces richer example sentences. This honours PRD §5 ("no server AI") — the summary is computed on the user's Mac through whatever engine they already chose.

Empty state

Fig 7 — Notebook empty state. A quiet illustration plus the one instruction that fills it. No "Add" button — the notebook fills itself from captures, so the empty state simply teaches the capture gesture.

2dPreferences window

A standard macOS toolbar-tab preferences window (now four tabs: General, Engine, Account, About), the only multi-tab window the app shows. Fixed width, height adapts per tab. The Account tab is new in rev 2 and is specified in §2e.

General

About

Fig 8 — Preferences ▸ About. App name, version/build, a GitHub link, the te CLI mention (PRD), license, and the on-device-OCR privacy line. The standalone "About" menu item shows this same content.

2ePreferences ▸ Account new · rev 2

The Account tab is where sync is turned on — and, just as importantly, where the app reassures users who never want an account. Its signed-out state leads with the "local is fine" message before the sign-in buttons, so signing in reads as an optional upgrade, never a wall. Its signed-in state shows the account, the sync toggle and last-synced time, and the two exits: Sign out (reversible) and Delete account & cloud data (destructive, confirmed).

Fig 9 — Account tab, both states. Signed out (left): the green "your notebook already works on this Mac… sign in only to sync across Macs" panel leads, then Sign in with Apple / Sign in with Google, then the text-only privacy line. Signed in (right): avatar + email + provider, a sync toggle, "Last synced…", Sign out (keeps local copy), and the destructive Delete account & cloud data (confirms first).

Detail — what "Delete account & cloud data" does

It is the only destructive control in the app, so it confirms with a sheet: "Delete your account and everything synced to the cloud? Your notebook stays on this Mac — only the cloud copy and your account are removed. This can't be undone." Buttons: Delete (red) / Cancel. It satisfies PRD §5/§10 (account + data deletion) and §6's privacy commitment. Sign out, by contrast, is non-destructive — it stops syncing and forgets the token but leaves both the local notebook and the cloud copy intact, so signing back in restores sync.

2fFirst-run onboarding

Three short steps in a single centred window, shown only on first launch (or when permission is missing). The middle step is the one that matters — it makes the Screen-Recording-then-relaunch dance painless.

Fig 9 — Onboarding. 1) Welcome + the default hotkey. 2) Screen Recording — explains why, opens the right Settings pane, and offers Quit & Reopen (relaunch is mandatory). 3) Done. On relaunch, a granted permission skips straight to step 3.

Detail — what step 2 detects

The app checks Screen Recording status (e.g. CGPreflightScreenCaptureAccess()) on launch. Granted → onboarding ends at step 1→3 with no friction. Not granted → step 2 is sticky. "Open System Settings" deep-links to Privacy & Security ▸ Screen Recording. After the user toggles the app on there, macOS itself prompts to quit; our "Quit & Reopen" does it cleanly so the user doesn't have to find us again. Polishing touch: if the user comes back without granting, the copy softens to "Still off — flip the switch next to Translator Everywhere."

3Brand & menu-bar status-icon concepts

App name — confirmed: "Translator Everywhere." It says exactly what it does (translate anything, anywhere on screen) and matches the PRD. Short-form in tight spots: "Translator". The menu-bar glyph must be a monochrome template image — single-colour, legible at 16–18 pt, auto-adapting to light/dark menu bars. Three concepts:

Fig 10 — Three template-glyph concepts, each shown large with a 16-pt menu-bar proof. All are single-stroke and silhouette-readable.

Recommendation — Concept A (speech-bubble + globe)

It reads as "translation" universally (the globe) and as "language/speech" (the bubble) without relying on a specific script, so it works for every locale and stays legible at menu-bar size. Concept B (譯/A) is the most literally "translation" but two glyphs get muddy at 16 pt and lean Chinese-specific; Concept C (crosshair-over-text) best describes the gesture but looks like a screenshot/measuring tool out of context. Ship A as the menu-bar icon; reuse a filled variant of A as the app/Finder/.dmg icon for a consistent identity. Take B or C as fallbacks if the PM prefers a more explicit "translate" mark.
