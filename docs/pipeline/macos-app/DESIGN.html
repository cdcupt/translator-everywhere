1Interaction model — the capture flow as states

One hotkey press drives a short-lived state machine. The user only ever sees three things: the native crosshair, then (briefly) nothing while we work, then the result panel. Every non-happy path resolves into a clear, dismissible panel — never a silent dead end, except a deliberate Esc-cancel, which returns to idle with no UI at all.

Fig 2 — The capture state machine. Happy path runs left→right then drops to result. Esc during selection returns to idle silently (dashed). No-text and error are recoverable panels. The permission gate diverts to onboarding step 2 before any capture starts.

⚑ The relaunch problem — handled in onboarding

Screen Recording permission only takes effect after the app restarts. So onboarding never tries to capture while ungranted. Step 2 detects the permission, opens System Settings to the right pane, and offers a "Quit & Reopen" button that relaunches the app for the user. On the fresh launch we re-check: granted → jump straight to step 3 ("Done — try it"); still not granted → stay on step 2 with a gentle "Still not enabled" hint. The hotkey is also live before permission is granted: pressing it while ungranted reopens onboarding step 2 rather than failing silently.

2aMenu-bar menu

The status item is a monochrome template glyph that adapts to light/dark menu bars automatically. Click it → this menu. It is the only always-available entry point.

Fig 3 — Menu-bar menu. The "Capture & Translate" row echoes the user's current hotkey so the binding is always discoverable. "Translate Clipboard" is the cheap should from PRD §7.

Behaviour notes. The hotkey label is data-bound to the recorded shortcut — change it in Preferences and the menu updates. If Screen Recording isn't granted yet, "Capture & Translate" still works but routes to onboarding step 2. "About" opens the same content as the Preferences About tab in a compact standalone panel.

2bResult popup

A small, focused NSPanel that appears near the captured region (or screen-centred as a fallback). Because agent apps launch unfocused, the panel explicitly becomes key and orders front so Esc and ⌘C work immediately. The translation is the hero; the source text is present but quiet.

Fig 4 — Result popup, success. Translation in the serif display face; recognized source dim below; "Copied ✓" confirms the silent clipboard write; the engine badge reads FREE or AI.

Fig 5 — Left: no-text-found (recoverable, Try Again re-arms the crosshair). Right: error — message names the real cause (offline / engine down / bad key) and offers Retry plus, for key/engine problems, Preferences. The recognized text is still copied where possible.

2cPreferences window

A standard macOS toolbar-tab preferences window (three tabs: General, Engine, About), the only "real" window the app shows. Fixed width, height adapts per tab.

Fig 6 — Preferences ▸ General. The hotkey field is the KeyboardShortcuts Recorder: click it, press a combo, done — never System Settings (PRD G2). Plus default direction and the should "Launch at login" toggle.

2dFirst-run onboarding

Three short steps in a single centred window, shown only on first launch (or when permission is missing). The middle step is the one that matters — it makes the Screen-Recording-then-relaunch dance painless.

Fig 9 — Onboarding. 1) Welcome + the default hotkey. 2) Screen Recording — explains why, opens the right Settings pane, and offers Quit & Reopen (relaunch is mandatory). 3) Done. On relaunch, a granted permission skips straight to step 3.

Detail — what step 2 detects

The app checks Screen Recording status (e.g. CGPreflightScreenCaptureAccess()) on launch. Granted → onboarding ends at step 1→3 with no friction. Not granted → step 2 is sticky. "Open System Settings" deep-links to Privacy & Security ▸ Screen Recording. After the user toggles the app on there, macOS itself prompts to quit; our "Quit & Reopen" does it cleanly so the user doesn't have to find us again. Polishing touch: if the user comes back without granting, the copy softens to "Still off — flip the switch next to Translator Everywhere."

3Brand & menu-bar status-icon concepts

App name — confirmed: "Translator Everywhere." It says exactly what it does (translate anything, anywhere on screen) and matches the PRD. Short-form in tight spots: "Translator". The menu-bar glyph must be a monochrome template image — single-colour, legible at 16–18 pt, auto-adapting to light/dark menu bars. Three concepts:

Fig 10 — Three template-glyph concepts, each shown large with a 16-pt menu-bar proof. All are single-stroke and silhouette-readable.

Recommendation — Concept A (speech-bubble + globe)

It reads as "translation" universally (the globe) and as "language/speech" (the bubble) without relying on a specific script, so it works for every locale and stays legible at menu-bar size. Concept B (譯/A) is the most literally "translation" but two glyphs get muddy at 16 pt and lean Chinese-specific; Concept C (crosshair-over-text) best describes the gesture but looks like a screenshot/measuring tool out of context. Ship A as the menu-bar icon; reuse a filled variant of A as the app/Finder/.dmg icon for a consistent identity. Take B or C as fallbacks if the PM prefers a more explicit "translate" mark.
