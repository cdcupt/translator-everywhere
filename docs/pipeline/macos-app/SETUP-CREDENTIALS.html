Setup · what the PM provides · Translator Everywhere

Three credential tasks, one by one

These unlock the cloud slices (accounts, sync, signing, deploy). Do them in any order — total ~30–40 min. When done, hand me the values at the bottom. The local-first app is being built in parallel and needs none of this.

Task 1 · ~15 min · developer.apple.com + appleid.apple.com

Apple — App ID, Sign in with Apple, signing

  1. Create the App ID. developer.apple.com → Account → Certificates, Identifiers & Profiles → Identifiers → ✚. Choose App IDs → App. Description: "Translator Everywhere". Bundle ID: Explicit = com.cdcupt.translator-everywhere. Under Capabilities, tick Sign in with Apple. Register.
  2. Get your Team ID. Account → Membership details → copy the 10-character Team ID.
  3. Create a Developer ID Application certificate (for notarizing a non-App-Store app). Easiest path: Xcode → Settings → Accounts → (your team) → Manage Certificates → ✚ → Developer ID Application. It installs into your login keychain. (Portal alternative: Certificates → ✚ → Developer ID Application, upload a CSR from Keychain Access.)
  4. Make an app-specific password for notarization. appleid.apple.com → Sign-In and Security → App-Specific Passwords → ✚, label it "notarytool". Copy it once.
Produces → App ID com.cdcupt.translator-everywhere · Team ID (10 chars) · a "Developer ID Application: …" cert in your keychain · an app-specific password.

Note: the native macOS Sign-in-with-Apple flow verifies an identity token on our server with Apple's public keys — no Apple private key / Services ID needed for v1. (If we later add server-side token refresh, you'd add a Sign-in-with-Apple Key → .p8; not now.)

Task 2 · ~10 min · console.cloud.google.com

Google — OAuth client for sign-in

  1. Project. console.cloud.google.com → create (or pick) a project, e.g. "Translator Everywhere".
  2. OAuth consent screen. APIs & Services → OAuth consent screen → External → fill app name + your support email. Scopes: just openid, email, profile. Add your own Google address under Test users (keeps it in testing — fine for us; no Google review needed).
  3. Create the client. APIs & Services → Credentials → Create credentials → OAuth client ID → Application type Desktop app → name it. Click create.
  4. Copy the Client ID (looks like …apps.googleusercontent.com). The "client secret" Google shows for a Desktop app is not confidential (the app uses PKCE) — send it too, I'll treat it as non-secret config.
Produces → a Google OAuth Client ID (Desktop app) + its client value.
Task 3 · ~5 min · your DNS provider for daichenlab.com

DNS — the backend subdomain

  1. Open the DNS for daichenlab.com (the same place you manage api.english.daichenlab.com / status.daichenlab.com).
  2. Add a record for api.translator that points to the same BWH box as your other daichenlab app subdomains — mirror the exact settings of the existing api.english.daichenlab.com record (same record type / IP / proxy on-or-off). That guarantees Caddy can issue TLS the same way it already does.
  3. Save. (I'll add the Caddy snippet + deploy the service on BWH myself once DNS resolves.)
Produces → api.translator.daichenlab.com resolving to BWH, ready for TLS.
Security: the app-specific password and any client value are secrets-ish — don't paste full secrets into chat. Put them in a provider store (like your ~/.openai/keys.env pattern, e.g. ~/.translator-everywhere/deploy.env on your Mac / the VPS). In chat, the Client ID, Team ID, bundle ID, and a "DNS is live" confirmation are fine — they're identifiers, not secrets.

Hand back to me when ready

  • ✅ App ID created (com.cdcupt.translator-everywhere) + your Team ID
  • ✅ Developer ID Application cert installed (just say "done") + the app-specific password stored in a deploy.env (not chat)
  • ✅ Google OAuth Client ID (the …googleusercontent.com string)
  • ✅ DNS live for api.translator.daichenlab.com ("done")
