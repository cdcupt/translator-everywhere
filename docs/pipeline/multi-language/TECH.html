1 Architecture — two surfaces, one catalog, no server

The change is contained to the client tier. The macOS app and the te bash CLI are independent codebases that must never drift on language codes, so they both derive from a single canonical languages.tsv. Translation still goes direct to Google / OpenAI — the sync server is not on this path and is not modified.

Why TSV-as-source (reconciled conflict)

Both workers proposed generation in opposite directions. Resolved in favor of a language-agnostic languages.tsv at the repo root: the Swift generator emits an embedded LanguageCatalog+Generated.swift (so the shipped .app needs no runtime file), while te reads the same TSV at runtime. A CI step re-runs the generator and git diff --exit-codes the generated Swift — the two surfaces cannot silently diverge.

2 Data model — the Language catalog & settings

One value type generalizes today's two-case LanguageDirection.Target (TranslationEngine.swift:81-92). Each field feeds a specific consumer; aiName is the single AI-capability source (nil = can't serve), exposed as a computed aiSupported.

TSV columns & the wire/request/result structs

languages.tsv — 6 columns

  • code — canonical BCP-47, the persisted value
  • englishName · endonym — labels + search
  • googleCode — real field (diverges: he/iw, jv/jw, zh-CN/zh-TW)
  • aiName — OpenAI hint; empty cell ⇒ nil ⇒ via-Google
  • aliases — extra search terms

Engine boundary structs

  • TranslationRequest{ text, from:Language?, to:Language }
  • TranslationResult{ translation, detected:DetectedSource, servedBy:EngineKind, viaGoogleFallback:Bool }
  • ResolvedEngine{ engine, viaGoogleFallback } from resolve(for:)
  • viaGoogleFallback is explicit, not inferred from servedBy

3 Source detection + same-language guard

Detection is Google-owned and engine-independent: the keyless translate_a/single response already carries the detected source at root[2] (fallback root[8][0][0]), mapped back via language(forGoogleCode:). Two physical paths, one logical contract. The same-language guard — a pure PairResolver — is what preserves EN⇄ZH (PRD S5).

Sequence — Auto path (Google one-call vs AI detect-first)

Flowchart — the guard decision

Edge handling

Very short / mixed input → low Google confidence (empty root[2] or near-zero root[8]) → DetectedSource.uncertain: render "Detected: —", pre-highlight the best guess in the From picker, and suppress the guard (translate to to as chosen) so a wrong flip is impossible. If root[2]'s shape shifts, fall back to root[8][0][0], else treat as uncertain — translation still succeeds.

4 Unsupported-pair "via Google" fallback

PM default #4: when the AI engine can't serve a pair, silently fall back to Google and surface a small "via Google" badge note — never a dead end. The signal is authoritative: the resolver computes viaGoogleFallback explicitly, so the UI never has to guess (and can distinguish a plain no-key FREE from an AI-routed-to-Google result).

resolve() lifecycle (reconciled)

Add resolve(for pair:) -> ResolvedEngine; keep the existing pair-agnostic resolve() (EngineResolver.swift:42-49) because summarize (NotebookView.swift:204-214) is pair-independent. The backend's "replace resolve()" becomes "add alongside".

5 Frontend — From/To picker, ResultPanel & state

The result surface is hand-rolled AppKit (ResultPanel is an NSPanel, ~440pt). Three new @MainActor components add a language bar above the translation block and a searchable popover picker. The capture state machine keeps its shape; multi-language adds a Picking → Re-translating loop and three conditions that decorate Success.

6 The te CLI — flags, config & parity

The CLI reaches parity with the app by reading the same languages.tsv (resolved next to the real script via realpath/readlink, since install.sh:14 symlinks te onto PATH). One ~/.te/config retargets every surface — including the three quick-action workflows, which pass no -t and inherit TE_TO.

Flags

FlagDefaultBehavior
-f · --from <code>$TE_FROM → autoSource language; validated against the TSV. Explicit From disables the guard.
-t · --to <code>$TE_TO → zh-CNTarget language; validated against the TSV. Unknown code → die with a hint to run te -l.
-l · --languages [query]—List the catalog as code · englishName · endonym; optional live awk filter.
--detect—Print the detected source (reuses the keyless Google detector) then exit; no translation.
-g / -a · --engineexistingEngine selection — orthogonal to From/To, preserved unchanged.
-e / -i / -c / --popup / -mexistingExplain / interactive / image / popup / model — unchanged; -e stays AI-only, forced off under -g.

Config — ~/.te/config

VariableDefaultNotes
TE_FROMautoSourced like the key file (set -a; . file; set +a) via a new load_config().
TE_TOzh-CNThe one place that sets the target for every CLI + quick-action surface.
TE_SECONDARYenBash same-language guard target when from=auto & detected==TE_TO.
TE_ENGINE · TE_MODEL—Optional; may also live here.

Precedence: explicit flag > env > ~/.te/config > built-in default (auto / zh-CN / en).

# default EN⇄ZH preserved — no flags $ te "Hello world" → 你好，世界 $ te "你好世界" → Hello world (guard: detected zh==TE_TO → TE_SECONDARY) # explicit pair + detect + list $ te -t fr "hello world" → bonjour le monde $ echo 中文 | te -g -f zh-CN -t ja → 中国語 (sl=zh-CN, tl=ja) $ te --detect "Bonjour le monde" fr $ te -l span Spanish · Español · es # one config retargets every surface, incl. quick actions $ cat ~/.te/config TE_TO=de $ te "hi" → hallo # "Translate with te" Service (te --popup) now shows the German dialog

Bash detection & guard (mirror of the app)

Google path reads the detected source from the same response with jq -r '.[2] // empty' (confirmed root[2]), printed dimmed (detected: fr). When from=auto and detected == $TE_TO, re-run with tl=$TE_SECONDARY. The AI path's hard-coded EN/ZH system prompt becomes parameterized — "Translate from <from-name | the detected source> into <to-name>" — with names mapped from the TSV.
