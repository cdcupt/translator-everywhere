Multi-Language Support
for Translator Everywhere

Lift the app and the te CLI out of English⇄Chinese and turn them into true any-language translators — a Google-Translate-style From / To experience across ~100+ languages, on every surface.

★ At a glance

The whole feature on one screen: the translation loop (unchanged shape), the one thing that changes inside it (the language model), and the scope of where it lands.

1 Problem & opportunity

Translator Everywhere is a fast, one-tap macOS translator (type / clipboard / screen-OCR → instant translation) plus a terminal te command. Both are locked to English ⇄ Simplified Chinese: the source is guessed by counting Han characters and the result simply flips to "the other one."

The engines are not the limitation — the free Google endpoint already accepts any tl= target, and the OpenAI engine can be prompted into any language. The limitation is the language model and UI: a 2-case Target enum, a binary flip, and no way to say "translate this into French." Any user whose work touches a third language is shut out of an otherwise excellent tool — and "it only does Chinese" caps the audience of a shipped, App-Store-style product.

Opportunity: with a real language catalog and a Google-style From / To picker, the same one-tap product serves the whole world's language pairs at near-zero added engine cost.

3 Goals & non-goals

✓ Goals

  • Translate to and from ~100+ languages (Google-style catalog), on both engines.
  • From / To pickers on the result surface — From: Auto ▾ + To: ▾, switchable per translation, with a searchable list.
  • Real source detection (Auto) replacing the 2-way Han flip; the detected language is shown.
  • Remember the last-used pair and offer a sensible default target.
  • Parity: the te CLI gains the same any-pair capability.
  • Preserve the one-tap feel — defaults must keep the zero-config fast path working with no extra clicks.
  • Notebook / study-list and the quick actions keep working with the new model.

✗ Non-goals (this round)

  • No changes to the sync server — translation never routes through it.
  • No new engines (still OpenAI + Google free).
  • No offline / on-device translation.
  • No document / file / batch translation.
  • No live conversation or speech mode.
  • No per-app or per-website language rules.

5 Product decisions (PM-confirmed)

Captured at scoping, before this PRD — these lock the design direction.

SurfaceApp + te CLI, at parity. Both client surfaces get multi-language; the sync server is untouched.
UX modelFull From / To pickers, Google-style. Two controls on the result surface — From: Auto ▾ and To: ▾ — switchable per translation, not a single fixed setting.
BreadthFull ~100+ language list. Everything the engines reasonably support, presented through a searchable picker (so a long list stays usable). OpenAI quality on rarer languages is a known, accepted variance.

6 Success criteria

Acceptance is binary per item — these become the QA test plan's spine and the beta Feature-Coverage Matrix.

S1From any of the ~100+ languages, a user can translate into any other, on both engines, on both surfaces.
S2From: Auto correctly detects the source for a spread of languages and shows what it detected.
S3The To picker (and From) is searchable and usable with a 100+ list; selecting a language re-translates without friction.
S4The last-used pair is remembered across launches; first-run has a sensible default that preserves today's one-tap behavior.
S5EN⇄ZH still works exactly as before for existing users (no regression in the default experience).
S6Notebook save / study-list and all quick actions function under the new model.
S7te exposes From/To via flags + config and reaches parity with the app; quick actions honor the target.
S8Ships cleanly: app builds + auto-updates via Sparkle; CLI installs; no broken existing paths.

7 Constraints, risks & assumptions

  • Engine quality variance. OpenAI handles common languages well but is uneven on rare ones; Google covers ~130 reliably. The picker offers the union; we accept the variance and may badge/limit where one engine can't serve a pair.
  • Language-code mapping. Google uses BCP-47-ish codes (zh-CN, pt-BR…); OpenAI takes a natural-language hint. A single canonical Language catalog must map cleanly to both. Single source of truth shared in spirit by app + CLI.
  • Auto-detect is now real. The 2-way Han flip is replaced by genuine source detection (Google returns the detected source; OpenAI can be prompted). Edge cases (mixed-language input, very short input) need defined behavior.
  • Preserve zero-friction. The product's value is one-tap speed. Adding pickers must not add a required step — defaults + memory carry the fast path.
  • Two codebases. Swift app and bash CLI must stay conceptually consistent (same languages, same defaults) without literally sharing code.
  • Privacy unchanged. Translation continues to go direct to the engine, never through TE's server.
  • Assumption: the free Google endpoint remains available and keyless, as today.
