1 Interaction model — pickers as defaults, never a step

The product's value is one tap → instant translation. The single most important decision in this design: the From / To pickers are passive controls that live on the result surface and only matter after a result already exists. Capture (or clipboard / CLI) fires immediately against the remembered last-used pair — no language is ever chosen up front, so the fast path keeps zero added clicks (PRD S4, S5). The pickers are a re-translate affordance: change To, the same text re-translates in place. The 100+ list is never in the way until you reach for it.

✓ What stays one-tap

  • Hotkey → OCR → result, using last pair. No picker shown first.
  • First run defaults reproduce today's EN⇄ZH (see §4).
  • Clipboard / CLI / quick actions: same — translate first, adjust after.

✗ What we deliberately avoid

  • No "pick a language" gate before the first translation.
  • No modal language sheet that blocks the result.
  • No always-expanded 100-row list crowding a 440px panel.

Design principle

The result is shown first; the language bar sits above it as a quiet, tappable summary (From: Auto ▾ · ⇄ · To: 中文 ▾). Speed is the default; control is one tap away.

2 Result-panel redesign

Where the controls sit inside the existing ResultPanel (an NSPanel, ~440pt wide). One new row — the language bar — is inserted between the header (badge / Copied / Save) and the Translation caption. Nothing else moves. The engine badge stays exactly where it is.

Layout rules (so a small panel stays uncrowded)

  • Single row. From · ⇄ · To on one line. If width is tight, the From/To controls show the endonym or code (中文, es) and full names live in the picker — they never wrap to two lines.
  • Detected line appears only when From = Auto and a translation has run, as a dim italic caption directly under the bar — never a third control.
  • Re-translate is in-place: the panel keeps its position; only the Translation block updates, with a brief inline spinner on the To control (no full reload, no flash).
  • Badge unchanged — AI (purple) / FREE (blue) keep their meaning and position; the badge can gain a small "via Google" note only in the pair-fallback state (§5).

3 The searchable picker — the make-or-break UX

Tapping From ▾ or To ▾ opens a popover anchored to that control (not a modal). A search field is auto-focused; the list shows Recent first, then the full A–Z catalog; typing filters live across English name, endonym, and code. This is how 100+ languages stay usable (PRD S3).

KeyBehaviour
typeFilters live; the list collapses to matches, first match highlighted. Matches on English name, endonym, and BCP-47 code.
↑ / ↓Move highlight through the filtered list (Recent + All), wrapping at the ends.
↩ ReturnSelects the highlight, closes, and re-translates. On an empty query, keeps the current language.
escCloses with no change (panel and result untouched).
clickSame as Return for that row.

Recent holds the user's home + secondary language and the last 3–5 chosen targets (most-recent first), so the 100+ list is rarely scrolled. Native NSPopover + a filtered table view; no custom window.

4 Defaults, memory & the swap — preserving EN⇄ZH

The subtle case: today "auto" is a bidirectional flip (Chinese→English, else→Chinese). A naïve fixed To: 中文 would translate captured Chinese into Chinese. To honour S5 exactly we keep a small same-language guard tied to a home / secondary pair.

BehaviourRule
First-run defaultFrom: Auto → To: 中文 (Simplified), secondary English. Reproduces today's experience for existing users with no setup (S4, S5).
MemoryThe last-used From/To pair persists across launches in SettingsStore (UserDefaults) and is the pair every fresh capture/clipboard/CLI call uses. Picking a new To updates the memory.
Same-language guardOnly with From: Auto. If the detected source equals the To language, translate into the secondary instead — so EN⇄ZH stays a flip. With an explicit From, no guard (the user asked for that pair).
Swap ⇄Exchanges From and To. When From: Auto, swap promotes the detected source into From (so Auto(ja) ⇄ 中文 → From: 中文 → To: Japanese) and re-translates. Disabled until a detection exists.

6 Preferences — inline vs. settings

Per-translation choice stays inline on the panel. Preferences only sets the home pair — the default target and secondary used by the fast path and the same-language guard. One small new section in the existing four-tab window (General · Engine · Account · About); the natural home is a Languages group on the Engine tab (or a new "Languages" tab).

Why so little in Prefs: the From/To pickers are the control surface. Duplicating them in Preferences would split state. Prefs answers one question — "when I capture without choosing, where does it go?"

7 The te CLI — parity in spirit

The CLI gains From/To via flags + env, mirroring the app's model: auto source by default, a remembered/configured target, the same language codes, and the same same-language guard. Quick actions read the configured target so they honour it too (PRD S7).

Flags & config

FlagMeaningDefault
-f, --from <code>Source language; auto = detect.$TE_FROM → auto
-t, --to <code>Target language (BCP-47, e.g. fr, zh-CN).$TE_TO → zh-CN
-l, --languages [q]List the catalog (code · name · endonym); optional filter q.—
--detectPrint only the detected source of the input, then exit.—
Env / configMeaningDefault
TE_FROMDefault source.auto
TE_TODefault target (the "home" language).zh-CN
TE_SECONDARYUsed by the Auto same-language guard.en

Config is sourced like the existing key file — add ~/.te/config (sourced the same way as ~/.openai/keys.env) exporting TE_TO / TE_FROM / TE_SECONDARY, so terminal use and the quick actions share one default. Codes are the same BCP-47 strings the app uses (§9).

# default: auto-detect → home (zh-CN), with EN⇄ZH guard preserved
$ te "Bonjour le monde"
detected: fr
你好，世界

# explicit target; -f auto stays implicit
$ te -t fr "hello world"
bonjour le monde

# explicit pair, free engine, piped input
$ echo "中文" | te -g -f zh-CN -t ja
中国語

# discover codes
$ te -l span
es Spanish Español
es-419 Spanish (Latin Am) Español (Latinoamérica)

Auto detection & the three quick actions

  • Google engine: already sends sl=auto; we now read the detected source from the response and print it dimmed (detected: ja), and send the chosen tl=<to>.
  • AI engine: the system prompt becomes parameterized — "Translate from <from|the detected language> into <to-name>" — replacing the hard-coded EN/ZH prompt. --detect asks the model only for the source label.
  • Same-language guard: when from=auto and detected == TE_TO, target TE_SECONDARY — the bash mirror of the app rule, so te on Chinese still gives English.
  • Quick actions: "Translate with te (AI)", "(Google)", and "screen region" all call te with no explicit -t, so they inherit TE_TO from ~/.te/config — one place to set the target for every surface. No per-workflow editing needed.

9 Language catalog — one shape, two surfaces

For the tech build: a single canonical catalog, mapped to both engines, shared in spirit by app and CLI (PRD constraint — no literal code sharing). Recommendation: keep the data as one source file and generate the bash list from it so they can't drift.

FieldPurposeExample
codeCanonical BCP-47 id; the value persisted & passed around.zh-CN, pt-BR, es
englishNamePicker label + search key.Chinese (Simplified)
endonymNative name shown + search key.中文（简体）
googleCodeValue for Google's tl= (often == code).zh-CN
aiNameNatural-language hint for the OpenAI prompt."Simplified Chinese"
aliases[]Extra search terms (autonyms, common spellings).["mandarin","zh","简体"]

This generalizes today's LanguageDirection.Target (macos/Sources/Translate/TranslationEngine.swift:81): the two-case enum becomes a Language value + a static catalog; googleCode already exists there and carries over directly. Auto is represented as the absence of a From Language (a sentinel), detection fills in the real source.

10 Open questions for the PM

Q1Same-language guard — confirm the model. To keep EN⇄ZH a flip without breaking "any source → 中文", I propose a home + secondary pair with a guard (§4). Alternative: drop the guard and make Auto→中文 literal (simpler, but capturing Chinese with the default would give Chinese→Chinese). Recommend the guard.
Q2Home default = 中文 or follow system locale? I default the home target to Simplified Chinese to match the current product and Erik's use. We could instead seed it from the macOS system language on first run. Recommend 中文 for zero-regression, switchable in Prefs.
Q3Picker placement in Preferences: a "Languages" group on the existing Engine tab, or a new dedicated Languages tab? Recommend a group on Engine to avoid a five-tab window.
Q4Unsupported-pair behaviour: silent fallback to Google with a "via Google" badge note (recommended, never a dead end) vs. an explicit "AI can't do this pair — use Free?" prompt. Recommend silent fallback.
