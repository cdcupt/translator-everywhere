OpenAI-key sync — login-only, encrypted at rest (Design)

PM: Erik · Coordinator: Claude · rev 2: PM chose login-only convenience over end-to-end encryption — no passphrase; the server can decrypt. Supersedes the E2E/passphrase design.

★ At a glance

out of the DB (so a DB-only leak is useless), never log the plaintext, TLS in transit, honest UI copy. Residual: a full-server breach (DB + env) can read keys; users can rotate the key at OpenAI.
Login-only sync: upload over TLS → encrypt at rest with a server master key → auto-restore on the next signed-in device. Simpler than E2E (no passphrase, no client crypto), at the cost that the server can decrypt.

1 · What changed & why (rev 2)

Rev 1 was end-to-end encrypted: the key was encrypted on the device under a user passphrase the server never saw. The PM decided that a remembered passphrase is unacceptable UX and that logging in should be enough to get the key back on a new Mac.

That is a deliberate, unavoidable trade: if login alone recovers the key, the server (which grants the login) must be able to recover it too — so this design is NOT end-to-end encrypted; the server can decrypt the key. We make it the responsible version of that choice (encrypt at rest, master key isolated from the DB, no plaintext logging, honest copy) rather than storing the key in the clear.

Accepted liability (PM sign-off): the OpenAI key is a paid bearer credential now stored on our server in a form the server can decrypt. A breach that obtains both the database and the master key can read users' keys. This is materially riskier than rev 1 and than today's local-only storage; it is accepted for the UX. Users can revoke/rotate the key at OpenAI if ever exposed.

2 · Encryption at rest (server-side)

AspectChoiceNotes
CipherAES-256-GCM (Go crypto/aes+cipher/gcm)Fresh random 96-bit nonce per write; 128-bit tag; AAD = user_id + "openai-key" so a row can't be replayed under another user.
Master keySECRET_ENCRYPTION_KEY — 32 random bytes, base64Lives only in ~/.translator-everywhere/deploy.env on BWH (like GOOGLE_CLIENT_SECRET), loaded at boot. Never in the repo, never in the DB. This isolation is what makes a DB-only leak useless.
What's stored{ nonce, ciphertext, tag } in user_secrets.blob (bytea)The plaintext key exists on the server only transiently, in memory, during PUT (encrypt) and GET (decrypt) — then discarded.
No client cryptoApp sends/receives the key as-is over TLSNo passphrase, no KDF, no client-side encryption. The app's only job is the Keychain + the toggle + the calls.
RotationMaster-key rotation = decrypt-all + re-encrypt (documented runbook)Rare; a versioned key_id byte in the blob lets old + new master keys coexist during rotation.
Why AES-GCM + a separate master key (not plaintext, not a KMS): on a single VPS there's no managed KMS, so the pragmatic best is a strong symmetric key held in the process env, separate from the database. That defeats the most common real leak — a stolen DB backup or a SQL-injection dump — because the ciphertext is useless without the env key. It does not defend a full-box compromise; see §6.

3 · Flows & states (much simpler now)

Enable — Mac A

  • Toggle "Sync this key across my Macs" under the key field (Engine tab). Disabled if not signed in / no key, with a one-line reason.
  • On → PUT the key (authenticated, TLS); confirm 2xx before showing "Synced". No sheet, no passphrase.
  • Editing the key later re-uploads silently; toggling off deletes the server copy.

New device — Mac B

  • After sign-in, the app GETs the key. If the server has one and this Mac has no local key → auto-restore to the Keychain (a small, non-blocking "Restored your OpenAI key" confirmation).
  • If this Mac already has a different local key → don't clobber: show a one-time "Use this Mac's key or the synced one?" (default keep local).
  • No key on the server → nothing happens; the field stays empty. (Never wipe a local key because the server is empty.)
No passphrase, no sheets, no unlock ceremony — this is exactly the "just log in and it's there" UX the PM asked for. The only new control is the sync toggle.

4 · Backend contract

Method · pathBody / returnsNotes
PUT /secret/openai-key{ key:"sk-…" } → 200 {updated_at}Server encrypts at rest; last write wins on updated_at. Size cap; TLS-only.
GET /secret/openai-key→ 200 {key:"sk-…", updated_at} · 404Server decrypts and returns to the authenticated owner only.
DELETE /secret/openai-key→ 204Disable / key-clear. Account-delete cascades automatically.
Registered in the existing authed chi group (session JWT). Server must: never log the key or blob (no body logger on /secret/*); never return another user's row; keep the plaintext only in-memory transiently; enforce a size cap; rate-limit. The key is plaintext in transit under TLS and briefly in server memory — inherent to this model, minimized and never persisted in the clear.

Truth-in-UI is a hard acceptance criterion: no on-screen statement may be false after this ships. The sync-ON line explicitly says we can decrypt it — no overclaiming "we can't read it."

6 · Threat model (honest)

Defended

  • DB-only leak (stolen backup, SQL-injection dump, co-tenant DB read): ciphertext is useless without the env master key.
  • Network interception: TLS; and the stored form is ciphertext.
  • Cross-user row replay: AAD binds user_id.
  • Casual/curious DB access without the app env.
  • Local key never lost: server-empty never wipes it.

NOT defended (accepted)

  • Full-server breach (DB + the process env/master key) → keys decryptable. This is the core accepted risk of login-only.
  • Malicious/compelled operator → can decrypt (by design).
  • Server memory scrape during PUT/GET → transient plaintext.
  • Stolen session JWT → attacker can GET the key (short-lived tokens + rate-limit reduce, don't remove).
Compared to rev 1 (E2E), this design gives up "a breach can't leak keys." That was the original ask; the PM has consciously traded it for zero-friction UX. Mitigations make it the safe-as-possible version of that choice, not a substitute for E2E.
