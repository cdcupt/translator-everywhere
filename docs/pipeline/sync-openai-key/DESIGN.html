★ At a glance

never hit the network/server. • Editing the key re-encrypts + re-uploads silently (cached KEK). • A breached server / DB dump → only the opaque envelope. • Forgotten passphrase → re-enter the key (30s); local key never lost. Security floor = passphrase entropy Argon2id throttles a GPU rig to ~10²–10⁴ guesses/s. App-generated 6-word / 128-bit secret ⇒ astronomically uncrackable. A weak human password would not be → gated.
Passphrase → Argon2id(+salt) → HKDF → KEK → AES-GCM(nonce, identity-bound AAD) → opaque envelope on the server. The passphrase is the only thing that unlocks it, and it never leaves the device.

1 · Cryptographic scheme (exact)

Primitives

StepChoiceParameters / notes
Passphrase → key (KDF)Argon2id (libsodium crypto_pwhash, MODERATE)m=256 MiB, t=3, p=1, 32-byte output. Per-user 16-byte random salt. Runs only on enable / unlock / change — never per request (~0.5–1 s on a Mac). Client enforces a params floor (reject a blob claiming < the floor).
Sub-key derivationHKDF-SHA256 expand (CryptoKit)Expand the Argon2id output to the AEAD key with info="te:openai-key:v1" — purpose-binds the KEK so it can't be reused for a different secret slot.
Encrypt the key (AEAD)AES-256-GCM (CryptoKit AES.GCM)Fresh random 96-bit nonce per seal (AES.GCM.Nonce()); 128-bit tag; plaintext = the key's UTF-8 bytes (optionally fixed-padded to hide length).
Authenticated data (AAD)canonical, length-prefixed{ schemaVersion, userID, kdfAlgo+params, salt, purpose="openai-key", version } — reconstructed identically on encrypt + decrypt, not stored separately.
Why each piece: Argon2id's memory-hardness is what makes offline brute-force of a stolen blob infeasible (PBKDF2 is GPU-cheap → rejected). GCM is an AEAD, so a wrong passphrase (wrong key) or any tampering fails closed on decrypt — no separate password verifier is stored (storing one would aid an offline attacker). Binding userID + version into the AAD means a malicious/breached server cannot install another user's blob or roll back to an old (possibly-leaked) key — the tag won't verify under this device's identity.

The stored envelope (opaque to the server)

not an oracle
salt + kdf_params travel inside the blob so a brand-new device with only the download can re-derive the key from the passphrase. The server stores the bytes whole and interprets nothing.

2 · Passphrase model & strength policy load-bearing

The scheme's security collapses to passphrase entropy, so this is a security control, not UX flavor. Both the crypto and adversary lenses converged here:

  • Default = app-generated. Offer a generated 6-word EFF-Diceware passphrase (~77 bits) — pre-selected — shown once with "Save this in your password manager. We can't reset it." + a copy button.
  • "Use my own" is behind a strict gate: live strength meter, minimum length (≥ 12 chars / "4+ random words"), reject a bundled common/breached-password list (offline list; or HIBP k-anonymity if we add it later), and block submit below a real strength threshold. An unconstrained passphrase is never accepted.
  • A required acknowledgement checkbox ("I understand there's no recovery") gates the primary button.
  • The passphrase is never persisted, logged, transmitted, or put in a Codable model. After unlock the device caches only the derived KEK (+ salt + epoch) in the Keychain, so editing the key re-encrypts silently without re-prompting.

3 · The three ceremonies & their states

Three deliberate ceremonies (Set / Unlock / Change) move between OFF and ON; everything else (translating, editing the key) happens silently in ON without the passphrase.

Enable (create vault) — Mac A

  • Toggle "Sync this key across my Macs (end-to-end encrypted)" under the key field. Disabled if not signed in ("Sign in on Account…") or no key ("Add a key above first").
  • Flip on → set-passphrase sheet (generated default / own-with-gate / ack checkbox).
  • Foreground, confirmed: encrypt → PUT → read back & decrypt to verify before showing "Synced". Cancel → toggle reverts to OFF.
  • Never overwrite a populated server slot from a device that hasn't decrypted it (create vs join).

Unlock (join vault) — Mac B

  • After sign-in (or foreground sync) the app detects blob present AND no local key → shows the unlock sheet once with "Not now"; then degrades to a calm persistent Engine row: "An encrypted key is waiting. [Unlock…] or paste a key above."
  • Success → write the key to the local Keychain, cache the KEK. Wrong passphrase → fail closed; offer "Forgot it? Paste your key instead."
  • Existing local key + server blob differ → one-time choice "keep this Mac's key or unlock the synced one?", default keep local. Never silently overwrite.
Inert on failure (critical). A pulled blob is attacker-controlled input. A decrypt/tag failure shows an error and changes nothing — never deletes the local key, never re-enrolls, never overwrites the server. "Server has no blob" must never delete a working local key.

4 · Multi-device & conflict

  • Version counter, not clock LWW. The server holds a monotonic version; PUT carries base_version and succeeds only if it matches (else 409 with the current blob) — deterministic and clock-skew-proof (a refinement over the PRD's "last-write-wins").
  • Client keeps a local high-water version mark and refuses a blob with a lower version → blocks server rollback.
  • Passphrase epoch: blobs are stamped so a unlock failure after a remote change shows "the passphrase was changed on another Mac — enter the new one" instead of a cryptic "wrong passphrase."
  • The key blob is carved out of the notebook's blind LWW path entirely.

5 · Backend contract

no parse/decrypt/KDF • no body logging on /secret/* routes • version conflict → 409; per-user rate-limit

Registered in the same authed chi group as /vocab (reuses the session-JWT middleware), but a separate endpoint + table — the high-value secret stays isolated, separately size-capped, and easy to audit. Deletion: disable → DELETE; local key-clear → client also DELETEs; account-delete → ON DELETE CASCADE (no new code).

7 · Threat model & residual risks (post-hardening)

Defended

  • Breached/curious server, full DB dump, backups, logs, shared-VPS co-tenant → opaque envelope only.
  • Network MITM / TLS-terminating proxy → ciphertext beneath TLS.
  • Malicious-server blob swap / install another user's key → AAD binds userID → tag fails.
  • Version downgrade / rollback → AAD version + local high-water mark.
  • Wrong passphrase, tampered/empty blob → fail closed, local state intact.

Residual / accepted

  • Weak passphrase → offline brute-force. Mitigated by Argon2id + the app-generated default + the strength gate (the §2 control).
  • Forgotten passphrase → no recovery (PM-locked); 30-second re-entry, local key never lost.
  • Compromised endpoint (malware on the Mac) → out of any app's scope.
  • Blob leaks the key length (GCM) → optional fixed-size padding.
  • 3rd-party KDF dep (libsodium) supply-chain → pin + checksum; only crypto_pwhash used.

8 · Decisions for your Gate-1 sign-off

D1 — Accept a crypto dependency for Argon2id? Recommended: yes — add swift-sodium (libsodium) for Argon2id. The built-in PBKDF2 alternative is GPU-cheap and materially weaker for a human passphrase; the adversary lens says "do not ship" with PBKDF2. Cost: one pinned, widely-audited dependency.
D2 — Default to an app-generated passphrase (6-word Diceware, "save it — no reset"), with "use my own" only behind a strict strength + breached-password gate? Recommended: yes — it's the single highest-leverage control against the one real residual risk. The alternative (let users type any passphrase) is what makes weak-passphrase brute-force possible.
D3 — Version-counter optimistic concurrency instead of the PRD's "last-write-wins"? Recommended: yes — deterministic + clock-skew-proof + blocks server rollback, at trivial cost. (This is the only place the design refines a PRD decision.)
Also confirm (defaults are sensible, override if you disagree): opt-in lives on the Engine tab under the key (Account = read-only mirror); the unlock prompt on a new Mac shows once then degrades to a calm row (no nagging); disabling sync deletes the server blob but keeps the local key; the privacy-copy rewrite ships with the feature (gating).
