★ At a glance

The whole feature on one screen: upload-on-A → encrypt-at-rest → auto-restore-on-B after sign-in. No passphrase; the server can decrypt (that's the accepted trade for zero-friction UX).

1 · Problem & the decision

Translator Everywhere lets a user bring their own OpenAI API key (BYOK) for the OpenAI translation engine. Today that key lives only in the local macOS Keychain. When the user signs in on a second Mac, the key isn't there — they must dig it out of OpenAI and re-enter it on every machine.

The PM wants the key to follow the user across devices with nothing to remember: sign in on a new Mac and the key is simply there. That convenience has an unavoidable consequence — if logging in alone restores the key, the server (which grants the login) must be able to restore it too. So this feature is NOT end-to-end encrypted: the server can decrypt the key.

PM decision (made with the trade-off spelled out): accept that the OpenAI key — a paid bearer credential — is stored on our server in a form the server can decrypt, in exchange for zero-friction UX. The earlier E2E/passphrase design (rev 1) is superseded. We ship the responsible version of this choice, not naïve plaintext.

2 · Users & the job

  • Multi-Mac BYOK user (primary): uses the OpenAI engine, signs in to sync their notebook, owns 2+ Macs. Job: "set my key once and have it work on all my Macs — with nothing extra to remember."
  • Convenience-first user: wants "just log in and it works," like every other SaaS. Accepts that the provider stores their key.
  • Single-Mac / Free-engine user: unaffected — the feature is opt-in and invisible unless they have a key and turn it on.

3 · Goals & non-goals

Goals

  • A signed-in user can opt in to sync their OpenAI key across their Macs.
  • On a new Mac, sign-in alone restores the key to the local Keychain and OpenAI works — no passphrase, no extra step.
  • The key is stored encrypted at rest, with the master key kept out of the database (a DB-only leak is useless).
  • The in-app privacy copy becomes accurate — it explicitly says we store the key and can decrypt it.
  • Zero impact on users who don't opt in.

Non-goals

  • End-to-end encryption / a user passphrase (explicitly traded away for UX).
  • Defending against a full-server compromise (DB + master key) — out of scope, accepted residual.
  • Syncing other secrets — only the OpenAI key ships now.
  • Server-side use of the key for translation — OpenAI calls stay on-device; the server only stores/returns the key.
  • Cross-ecosystem clients today (design stays portable, but only macOS ships).

5 · Success criteria (acceptance)

  • Round-trip: enter key + toggle sync on Mac A → sign in on Mac B → the exact same key is auto-restored to Mac B's Keychain (no passphrase) and a Test call succeeds.
  • Encrypted at rest: the stored row is ciphertext; the master key is not in the database. A DB-only dump yields no usable key. (Verified by direct inspection.)
  • No plaintext leaks to logs: the key never appears in server logs on PUT or GET. (Verified.)
  • Don't clobber: a new Mac that already has a different local key is not silently overwritten; the user chooses.
  • Never wipe: an empty server copy never clears a working local key.
  • Opt-out: disabling sync (or clearing the key, or deleting the account) deletes the server copy; the local key remains usable.
  • No-opt-in users see and store nothing new; existing local-only behavior unchanged.
  • Honest copy: no on-screen statement is false after the change; the sync-on copy explicitly says "we can decrypt it — not end-to-end."
  • Build green; automated tests cover encrypt/decrypt round-trip, wrong-master-key fails closed, auth (no cross-user read), delete; beta panel ships.

6 · Constraints & principles

  • Convenience is the point. Sign-in alone recovers the key; no passphrase, no ceremony. Everything else serves that.
  • Encrypted at rest, master key isolated. The server encrypts the key with a strong symmetric key held in the process env — never in the DB, never in the repo. This defeats the common DB-only leak.
  • Least plaintext exposure. The server sees the key only transiently in memory during PUT/GET; it is never logged or written in the clear.
  • Truth in UI. Every privacy claim shown to the user must be literally true after this ships — including that we can decrypt the key.
  • Fail safe on the client. Never overwrite a different local key without consent; never wipe a local key because the server is empty.
  • Opt-in, reversible, low-blast-radius. Off by default; turning it off deletes the server copy.
  • Rotatable blast radius. If a key is ever exposed, the user can revoke/rotate it at OpenAI — the residual harm is bounded.

7 · Threat model (honest — full detail in the Design Doc)

Full-server breach (DB + process env/master key) → keys decryptable. The core accepted risk of login-only. • Malicious / compelled operator → can decrypt (by design). • Server memory scrape during PUT/GET → transient plaintext. • Stolen session token → attacker can GET the key (reduced by short-lived tokens + rate-limit, not removed).
This design deliberately gives up "a breach can't leak keys" (rev 1's guarantee) for zero-friction UX. The mitigations make it the safe-as-possible version of that choice, not a substitute for E2E.
