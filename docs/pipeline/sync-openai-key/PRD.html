★ At a glance

passphrase never leaves the device → the server only ever holds ciphertext it can't read. A full server breach leaks no usable keys. The OpenAI key still goes only to OpenAI. In scope opt-in sync · passphrase set/enter · E2E crypto · backend ciphertext blob · multi-device LWW · honest privacy-copy rewrite Out of scope passphrase recovery · syncing other secrets · server-readable keys · iCloud Keychain · non-Apple clients (design stays portable)
The whole feature on one screen: encrypt-on-A → store-ciphertext → decrypt-on-B, with the passphrase as the only thing that can unlock it, and it never leaves the device.

1 · Problem & why now

Translator Everywhere lets a user bring their own OpenAI API key (BYOK) to use the OpenAI translation engine. Today that key lives only in the local macOS Keychain and is never sent to our server. That's privacy-safe but inconvenient: when the user signs in on a second Mac, the key isn't there — they must dig it out of OpenAI and re-enter it on every machine.

The PM wants the key to follow the user across their devices once signed in — without our service ever being able to read it. A naïve "upload the key to our VPS" is rejected outright: an OpenAI key is a bearer credential to a paid account, so a server breach would leak credentials that cost users real money, and it would contradict the app's own on-screen privacy promise. This feature delivers the convenience under a strict end-to-end-encrypted constraint.

2 · Users & the job

  • Multi-Mac BYOK user (primary): uses OpenAI engine, signs in to sync their notebook, owns 2+ Macs. Job: "set my key once and have it work on all my Macs."
  • Privacy-conscious user: will only opt in if they trust that we genuinely can't read the key. Job: "give me the convenience without trusting your server with my credential."
  • Single-Mac / Free-engine user: unaffected — the feature is opt-in and invisible unless they have a key and turn it on.

3 · Goals & non-goals

Goals

  • A signed-in user can opt in to sync their OpenAI key across their Macs.
  • On a new Mac, after sign-in + passphrase, the key is restored to the local Keychain and OpenAI works.
  • The server provably cannot read the key (it holds ciphertext only; the passphrase never leaves the device).
  • The in-app privacy copy becomes accurate for the new behavior.
  • Zero impact on users who don't opt in.

Non-goals

  • Passphrase recovery / "reset my passphrase and keep my key" (impossible under E2E — by design).
  • Syncing other secrets (auth tokens, other API keys) — the blob format is extensible but only the OpenAI key ships now.
  • Server-side use of the key (server never decrypts; OpenAI calls stay on-device).
  • Cross-ecosystem clients today (design stays portable, but only macOS ships).

5 · Success criteria (acceptance)

  • Round-trip: enter key + opt in on Mac A → sign in + enter passphrase on Mac B → the exact same key is in Mac B's Keychain and a Test call succeeds.
  • Server-blind: inspecting the server DB + all server logs reveals no recoverable key material — only ciphertext, salt, nonce, and KDF params. (Verified by direct inspection.)
  • Wrong passphrase on Mac B fails closed with a clear message and never corrupts or partially-restores the key.
  • Opt-out: disabling sync deletes the server blob; the local key remains usable.
  • No-opt-in users see and store nothing new; existing local-only behavior unchanged.
  • Honest copy: no on-screen statement is false after the change.
  • Build green, automated tests cover the crypto round-trip + wrong-passphrase + blob tamper; beta panel ships.

6 · Constraints & principles

  • E2E or nothing. The encryption/decryption key is derived from the user's passphrase on the device; it and the passphrase never touch the network or the server.
  • Server is a dumb, blind store. It authenticates the user (existing session JWT), enforces a size cap, stores/returns opaque bytes, and never logs or decrypts them.
  • Fail closed, never silently. Any decrypt/auth failure surfaces clearly and leaves existing state intact — no half-restored or corrupted key.
  • Local Keychain stays the fast path. A device that already has the key needs no passphrase for normal use; the passphrase is only for upload-on-opt-in and restore-on-new-device.
  • Truth in UI. Every privacy claim shown to the user must remain literally true.
  • Opt-in, reversible, low-blast-radius. Off by default; turning it off cleans up.

7 · Threat model (summary — full scheme in the Design Doc)

Weak passphrase → offline brute-force of a stolen blob. Mitigation: memory-hard KDF + a minimum-strength policy. • Compromised endpoint (malware on the user's Mac reading the Keychain or keystrokes) — out of any app's scope. • Forgotten passphrase → no recovery (accepted trade-off). • A maliciously-served client — N/A: we ship a signed native binary.
The design must make the one residual in-scope risk — offline brute-force of a weak passphrase — economically infeasible via the KDF + a strength policy. The exact KDF/params are a Gate-1 decision in the Design Doc.
