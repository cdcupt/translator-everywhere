★ At a glance

Encrypt at rest AES-256-GCM, master key in deploy.env not the DB, key_id byte for rotation, AAD binds user_id. ② Graceful-degrade boot: if the master key is absent/invalid the server still starts; only /secret/* returns 503 (avoids a shared-box 502 blast radius). ③ Auto-restore after sign-in when the local Keychain is empty; never clobber a different local key; never wipe on empty server. ④ Hardening: per-user rate-limit (30/min, 429), size cap 4 KB (413), no key/blob in logs, TLS-only. ⑤ Reuse existing patterns: SecretSyncClient mirrors SyncClient (401→refresh-once); append to existing files (no new Xcode files); +dep go-chi/httprate. The server can decrypt the key — intentional (login-only). Not end-to-end. Users can rotate the key at OpenAI.
The whole system on one screen: app ↔ TLS ↔ Go server ↔ Postgres, with the master key isolated in the process env; the two flows (enable→upload→encrypt, sign-in→fetch→decrypt→restore) run over the same spine.

2 · Backend / Infrastructure design (Go)

2.1 Data model

One optional secret row per user, keyed by (user_id, name). name is left unconstrained (YAGNI) but the handler hardcodes the literal "openai-key" — no caller-supplied name. Point lookup + upsert only, so the PK is the only index needed.

2.2 Encryption envelope & the Sealer

Package server/internal/secrets/ (new)

  • NewSealer(base64Key string) (*Sealer, error) — decodes + validates the master key is exactly 32 bytes; error otherwise (drives graceful-degrade at boot).
  • Seal(plaintext, aad []byte) ([]byte, error) — 0x01 ‖ rand12 ‖ gcm.Seal(...) using crypto/aes+crypto/cipher and crypto/rand.
  • Open(blob, aad []byte) ([]byte, error) — reads key_id, dispatches to the master key, GCM-opens; any failure returns an error and no plaintext (fail closed).
  • AAD is assembled in the handler (append(userID[:], []byte("openai-key")...)), not inside Sealer — keeps the crypto pure.

2.3 Endpoints

Verb · pathAuthBody / returnsCodes
PUT /secret/openai-keysession JWT{key} → {updated_at}; Seal + upsert LWW on updated_at200 · 400 (empty) · 413 (>4 KB) · 401 · 429 · 503 (no master key)

Auth invariant: user_id comes only from the verified JWT (userIDFromContext, middleware.go:46) — never from the body. A user can only ever read/write their own row; a GET for another user yields 404 because the row is keyed on their id.

2.4 Hardening

Secrets never leak

  • No body-logging middleware exists today (server.go stack = RequestID/RealIP/Recoverer); we add none for /secret/*. Error paths log "decrypt failed for user %s" — user_id only, never the key/blob.
  • Only the GET response body carries plaintext; PUT/DELETE responses never echo the key.

Abuse & size

  • Per-user rate limit 30/min (go-chi/httprate, keyed on JWT sub) → 429; scoped to the /secret/* subgroup only.
  • Body size cap 4 KB → 413 (a real key is ~a few hundred bytes).
  • TLS-only (Caddy edge).

5 · Test plan (QA)

How this repo tests

  • Backend: cd server && go vet ./... && go test -race ./... — stdlib testing + httptest + in-memory db.FakeRepository (CI .github/workflows/server.yml).
  • macOS: Swift Testing (@Test/#expect) via xcodebuild test (scheme "Translator Everywhere", platform=macOS, CODE_SIGNING_ALLOWED=NO); HTTP faked by MockURLProtocol; Keychain + sync-fake patterns already exist in macos/Tests/.
  • Manual-only (can't run headlessly): the two-Mac round-trip (AC1), live-session Keychain ACL, prod DB / deploy.env inspection, prod-log grep, rotated/wrong-master-key ops check.

Backend unit — crypto (secrets)

  • Seal→Open round-trips; two seals of same input differ (fresh nonce).
  • Wrong master key → error + empty plaintext (fail closed).
  • Tamper ciphertext / tag / nonce byte → GCM rejects, no plaintext.
  • AAD binding: blob for user A won't Open under user B.
  • Constructor rejects non-32-byte / non-base64 / empty key.
  • key_id rotation: old blob opens via keyring; unknown id → error, not panic.

Backend config / boot

  • Missing/short/non-base64 key → graceful degrade: server boots, /secret/* → 503 (Coordinator decision; QA asserts this, not fail-fast).
  • Valid key → routes live; key value never in any error/log string.

Backend integration (httptest + fake)

  • PUT then GET (same owner) → identical key + updated_at.
  • GET as a different user → 404 (no cross-user read).
  • PUT > 4 KB → 413; normal → 200.
  • PUT older→newer → LWW by updated_at; stale write ignored.
  • DELETE → 204; next GET → 404.
  • Account-delete cascades the secret row.
  • No plaintext in logs: logger→buffer, PUT/GET a sentinel key, assert absent — incl. forced error paths.
  • Unauthenticated → 401; over rate limit → 429.

Client unit (mock client + test Keychain)

  • enable → upload once with the key; edit-while-on → re-upload.
  • disable → delete called + local key retained.
  • auto-restore when local empty → key in Keychain + toast.
  • don't-clobber different local key → prompt, Keychain unchanged; same key → no-op.
  • never-wipe on 404/empty → local untouched.
  • toggle disabled when signed-out / no key (reason shown); signed-out → no network.

Copy / truth-in-UI

  • Assert the three DESIGN §5 strings verbatim; when sync ON, the stale "never written to disk…" string is absent and "can decrypt"/"not end-to-end" present.

6 · Technical decisions for Gate-2 sign-off

T1 — Graceful-degrade boot. If SECRET_ENCRYPTION_KEY is absent/invalid, the server still starts and only /secret/* returns 503; the rest (auth, notebook sync) is unaffected. Rationale: this is a shared multi-tenant box — coupling total server uptime to one new env var risks a 502 blast radius on every rollout. recommend: yes (safer than fail-fast)
T2 — Hardening limits. Per-user rate limit 30/min (429); body cap 4 KB (413); adds go-chi/httprate. recommend: yes
T3 — Auto-restore + confirmation surface. Auto-restore silently when the Keychain is empty; confirmation shown on the Account signed-in view + Engine status (no OS-notification permission). Opt-out is the toggle, not clearing the field. recommend: yes
Also confirming: reuse existing patterns (SecretSyncClient ⇄ SyncClient, append-to-existing-files); LWW on updated_at; name column unconstrained but handler-hardcoded to "openai-key"; container-only redeploy.
